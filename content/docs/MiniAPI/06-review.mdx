## Key Concepts

### 1. Type Safety First

**Bad: Everything as strings**

```python title="❌ Version 1 — no constraints"
class Match(BaseModel):
    sport: str    # accepts "fotball", "Soccer", "⚽", anything
    status: str   # accepts "done", "finished", "COMPLETED"
    date: str     # accepts "tomorrow", "next week", "asdf"
```

Users can send `"fotball"`, `"done"`, `"tomorrow"` — all accepted, all corrupting your data.

**Good: Proper types**

```python title="✅ Version 2 — constrained and validated"
class Match(BaseModel):
    sport: Sport    # !mark
    status: Status  # !mark
    date: date      # !mark
```

Invalid values are rejected immediately with clear error messages — no extra code needed.

<Callout type="info" title="Lesson">
  **Use the type system.** Don't accept strings when you want enums. Don't accept strings when you want dates. Let Pydantic do the work — that's exactly what it's designed for.
</Callout>

---

### 2. Pydantic Handles Types, You Handle Logic

| Pydantic Validates Automatically | You Must Validate Manually |
|----------------------------------|---------------------------|
| Is `status` a valid enum value? | Can `status=upcoming` have a winner? |
| Is `date` a real calendar date? | Is the date in the past for upcoming matches? |
| Is `sport` one of 5 options? | Are both teams actually different? |

<Callout type="warn" title="Lesson">
  **Type validation is automatic. Business logic validation is manual.**

  Pydantic ensures values are the right **type**. You ensure they make **logical sense** in your domain. Neither replaces the other.
</Callout>

---

### 3. Filter Contradictions

Some filter combinations are logically impossible:

```bash
# ❌ Impossible: upcoming match with a winner
GET /matches?status=upcoming&winner=home_team

# ❌ Impossible: team won but winner is draw
GET /matches?team_filter=won&winner=draw

# ❌ Impossible: winner filter on non-completed matches
GET /matches?status=cancelled&winner=home_team
```

Always validate filter combinations before processing:

```python title="matches.py"
def validate_filter_combinations(status, winner, team_filter):
    if team_filter and winner:  # !mark
        return {"error": "Cannot use both filters together"}

    if (team_filter or winner) and status != Status.completed:  # !mark
        return {"error": "These filters require completed matches"}

    return None  # No error — proceed with filtering
```

<Callout type="info" title="Lesson">
  **Fail fast with clear errors.** Don't return empty results for impossible queries — return an error explaining the contradiction. It's more honest and much easier to debug.
</Callout>

---

### 4. Two-Parameter Filter Pattern

For comparisons or qualified lookups, use two parameters:

**Date filtering:**
- `match_date` = the value to compare against
- `date_filter` = the operator (`eq` / `gt` / `lt`)

**Team filtering:**
- `team_name` = which team to search for
- `team_filter` = what outcome to look for (`won` / `lost` / `draw`)

```bash
# After a specific date
GET /matches?match_date=2026-01-01&date_filter=gt

# Team wins only
GET /matches?team_name=arsenal&team_filter=won
```

<Callout type="info" title="Lesson">
  **One parameter for value, one for operation.** This pattern appears throughout REST API design for dates, numbers, and other filterable fields. It keeps the API flexible without creating a parameter explosion.
</Callout>

---

### 5. Empty Results vs Errors

**Return empty results when:**

```bash
GET /matches?sport=hockey
→ {"message": "No matches found for the given filters"}
# Valid query — just no hockey data yet
```

**Return errors when:**

```bash
GET /matches?status=upcoming&winner=home_team
→ {"error": "winner filter requires completed status"}
# Invalid query — the combination is logically impossible
```

<Callout type="warn" title="Lesson">
  **Empty ≠ Error.** Valid queries with no results return empty responses (with a helpful message). Invalid queries return error messages. The distinction matters for the developers using your API.
</Callout>

## Production Considerations

The API you've built works well for learning, but production apps need more. Here's the roadmap:

### 1. Real Database

Right now, data is in-memory and **disappears when the server restarts**:

```python title="❌ In-memory — data lost on restart"
INITIAL_DATA: List[Match] = [...]
```

For production, use a real database:

```python title="✅ With SQLAlchemy + PostgreSQL"
from sqlalchemy.orm import Session

@app.get("/matches")
def list_matches(db: Session = Depends(get_db), status: Optional[Status] = None):
    query = db.query(MatchModel)

    if status:
        query = query.filter(MatchModel.status == status)

    return query.all()
```

<Callout type="info" title="What is Depends?">
  `Depends(get_db)` is FastAPI's **dependency injection** system. `get_db` is a function that opens a database session and closes it when the request is done. FastAPI calls it automatically and passes the result as `db`. This pattern keeps your route functions clean and testable.
</Callout>

---

### 2. Pagination

Large result sets need pagination — returning 10,000 matches at once would be slow and wasteful:

```python title="matches.py"
@app.get("/matches")
def list_matches(
    ...,
    page: int = 1,  # !mark
    page_size: int = 20,  # !mark
):
    # ...apply filters...

    total = len(matches)
    start = (page - 1) * page_size
    end = start + page_size

    return {
        "total": total,
        "page": page,
        "page_size": page_size,
        "matches": matches[start:end]   # Only return one page
    }
```

<Callout type="info" title="How pagination works">
  `page=1, page_size=20` means "give me items 1–20". `page=2, page_size=20` means "give me items 21–40". The formula `start = (page - 1) * page_size` converts a page number to an array index. Users call `?page=2` to get the next batch.
</Callout>

### 3. Proper HTTP Status Codes

Currently we return plain dicts for errors. Production APIs use HTTP status codes so clients know what happened without parsing the body:

```python title="matches.py"
from fastapi import HTTPException

@app.get("/matches/{match_id}")
def get_match_by_id(match_id: int):
    match = next((m for m in INITIAL_DATA if m.id == match_id), None)

    if not match:
        raise HTTPException(status_code=404, detail="Match not found")  # !mark
        # 404 = "Not Found" — the standard HTTP code for missing resources

    return match
```

<Callout type="info" title="Common HTTP status codes">
  - `200` — OK (success)
  - `201` — Created (POST success)
  - `400` — Bad Request (invalid input)
  - `404` — Not Found
  - `422` — Unprocessable Entity (Pydantic validation failure)
  - `500` — Internal Server Error

  Clients like mobile apps and JavaScript frontends check the status code first to decide how to handle the response.
</Callout>

### 5. Testing

A well-tested API gives you confidence to refactor and extend. Use `pytest`:

```python title="test_matches.py"
import pytest
from fastapi.testclient import TestClient
from matches import app

client = TestClient(app)

def test_filter_by_status():
    response = client.get("/matches?status=completed")
    assert response.status_code == 200
    data = response.json()
    assert all(m["status"] == "completed" for m in data)

def test_contradictory_filters_return_error():
    response = client.get("/matches?status=upcoming&winner=home_team")
    assert response.status_code == 200
    assert "error" in response.json()

def test_create_match_same_team_rejected():
    response = client.post("/matches", json={
        "home_team": "Arsenal",
        "away_team": "Arsenal",   # Invalid — same team
        "sport": "football",
        "date": "2026-05-01",
        "status": "upcoming"
    })
    assert "error" in response.json()
```

<Callout type="info" title="TestClient">
  FastAPI's `TestClient` lets you make HTTP requests to your app without running a real server. It's fast, and tests run in isolation. Write one test per rule — if you later change the validation logic, the tests will tell you immediately what broke.
</Callout>

### 7. Deployment with Docker

Package the app so it runs consistently anywhere:

```dockerfile title="Dockerfile"
FROM python:3.11-slim

WORKDIR /app

# Install Poetry
RUN pip install poetry

# Copy dependency files first (for Docker layer caching)
COPY pyproject.toml poetry.lock ./
RUN poetry install --no-root --no-dev  # !mark

# Copy app code
COPY matches.py .

CMD ["poetry", "run", "uvicorn", "matches:app", "--host", "0.0.0.0", "--port", "8000"]
```

```bash
docker build -t sports-api .
docker run -p 8000:8000 sports-api
```

<Callout type="info" title="Why copy dependency files first?">
  Docker builds images in layers. By copying `pyproject.toml` and `poetry.lock` before the app code, the `poetry install` layer is cached. If you only change `matches.py`, Docker skips reinstalling dependencies and the build is much faster.
</Callout>
