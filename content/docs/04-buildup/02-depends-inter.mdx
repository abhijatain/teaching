In the last chapter you learned that a dependency is a function FastAPI calls before your route. But if you look at the `list_matches` route, you've actually been using dependencies based on *classes* this whole time — and never knew it:

```python
@router.get("", ...)
def list_matches(
    filters: Annotated[MatchFilters, Depends()],        # ← class dependency
    date_range: Annotated[DateRangeFilters, Depends()], # ← class dependency
    pagination: Annotated[PaginationParams, Depends()], # ← class dependency
    sort_params: Annotated[SortParams, Depends()],      # ← class dependency
):
```

`MatchFilters`, `DateRangeFilters`, `PaginationParams`, and `SortParams` are all Pydantic classes used as dependencies. This chapter explains exactly how and why that works, then takes it a step further with sub-dependencies — dependencies that themselves depend on other dependencies.

By the end of this chapter you'll have:
- A solid understanding of how classes work as dependencies
- A `get_query_bundle` function that wraps all four query-param dependencies into one
- A `QueryBundle` model that groups all the query concerns, making `list_matches` cleaner

## How Classes Work as Dependencies

When you write `Depends(some_function)`, FastAPI calls that function and injects the return value. When you write `Depends()` with no argument — or `Depends(SomeClass)` — FastAPI calls the class's `__init__` method instead.

Calling `__init__` is just another way of saying "create a new instance." So `Depends(MatchFilters)` is FastAPI saying: "create a new `MatchFilters` instance, using the incoming request data to fill in the fields."

Here's what happens step by step when FastAPI processes `filters: Annotated[MatchFilters, Depends()]`:

```
Request arrives: GET /matches?sport=football&status=upcoming
                       │
                       ▼
  FastAPI sees: Annotated[MatchFilters, Depends()]
  → reads MatchFilters.__init__ (its field definitions)
  → sees: sport, status, winner, team, team_filter (all Optional with defaults)
  → extracts values from the query string
  → calls: MatchFilters(sport="football", status="upcoming")
  → Pydantic validates and normalises the values
  → model_validator runs
                       │
                       ▼
  filters = MatchFilters(sport=Sport.football, status=Status.upcoming, ...)
  injected into list_matches
```

The `Depends()` with no argument is shorthand for `Depends(MatchFilters)` — FastAPI infers the class from the type annotation. Both work identically:

```python
# These are equivalent:
filters: Annotated[MatchFilters, Depends()]
filters: Annotated[MatchFilters, Depends(MatchFilters)]
```

The first form is cleaner and more commonly used when the dependency type matches the annotation type exactly.

<Callout type="info" title="Why Pydantic models work so well as dependencies">
  A Pydantic model's `__init__` signature is auto-generated from its field definitions. FastAPI reads that signature the same way it reads any function — it sees the parameter names, types, and defaults, then extracts matching values from the query string.

  This is exactly why we switched from `Query()` to `Depends()` in an earlier chapter. `Query()` is for individual query parameters. `Depends()` on a class groups many query parameters together into a structured, validated object. The class's validators and `model_validator` run as part of the injection — all the validation you wrote in `MatchFilters` runs automatically before your route function even starts.
</Callout>

## Sub-dependencies — Dependencies That Depend on Other Dependencies

A sub-dependency is simply a dependency function whose parameters are themselves dependencies. FastAPI builds a dependency graph and resolves everything in the right order.

Look at the current `list_matches` signature:

```python
def list_matches(
    filters: Annotated[MatchFilters, Depends()],
    date_range: Annotated[DateRangeFilters, Depends()],
    pagination: Annotated[PaginationParams, Depends()],
    sort_params: Annotated[SortParams, Depends()],
):
```

Four separate parameters, all query-related. The function signature is getting long before the actual logic even starts. You can bundle all four into a single dependency using a sub-dependency:

```python title="app/routes/matches.py"
def get_query_bundle(
    filters: Annotated[MatchFilters, Depends()],          # !mark
    date_range: Annotated[DateRangeFilters, Depends()],   # !mark
    pagination: Annotated[PaginationParams, Depends()],   # !mark
    sort_params: Annotated[SortParams, Depends()],        # !mark
) -> QueryBundle:
    return QueryBundle(
        filters=filters,
        date_range=date_range,
        pagination=pagination,
        sort_params=sort_params,
    )
```

`get_query_bundle` is a function that takes four dependencies as parameters. FastAPI resolves each of those dependencies first (building `MatchFilters`, `DateRangeFilters`, etc. from the query string), then calls `get_query_bundle` with all four, then injects the returned `QueryBundle` into your route.

The route signature becomes:

```python title="app/routes/matches.py"
@router.get("", response_model=MatchListResponse, response_model_exclude_none=True)
def list_matches(q: Annotated[QueryBundle, Depends(get_query_bundle)]):  # !mark
    ...
```

One parameter instead of four. The function body accesses everything through `q`:

```python
# Before:
if filters.sport:
if date_range.from_date:
results[pagination.offset : pagination.offset + pagination.limit]

# After:
if q.filters.sport:
if q.date_range.from_date:
results[q.pagination.offset : q.pagination.offset + q.pagination.limit]
```

## The `QueryBundle` Model

`QueryBundle` is the container that `get_query_bundle` returns. Add it to `app/models.py`:

```python title="app/models.py"
class QueryBundle(BaseModel):
    """Combines all query concerns into a single injectable dependency."""

    model_config = {"arbitrary_types_allowed": True}  # !mark

    filters: MatchFilters
    date_range: DateRangeFilters
    pagination: PaginationParams
    sort_params: SortParams
```

The `model_config = {"arbitrary_types_allowed": True}` line deserves an explanation. By default Pydantic only allows "simple" types as field values — strings, ints, other Pydantic models it knows how to validate. When you nest one Pydantic model inside another as a field type, Pydantic needs to be told this is intentional. `arbitrary_types_allowed` tells Pydantic to trust that these nested models are already valid and not try to re-validate them from scratch.

<Callout type="info" title="When do you need arbitrary_types_allowed?">
  You need it when a Pydantic model contains fields typed as other Pydantic models that weren't created by parsing raw data. Here, `MatchFilters` etc. were already created and validated by FastAPI's dependency system — `QueryBundle` is just grouping pre-built objects together. Without this config, Pydantic would try to re-validate them as if they were raw dicts, which would fail.
</Callout>

## FastAPI's Dependency Graph — Only Called Once

Here's something important: if the same dependency appears multiple times in the resolution chain, FastAPI calls it only once per request and reuses the result.

Imagine both `get_query_bundle` and some other function both depended on `PaginationParams`. FastAPI would call `PaginationParams(...)` once, cache the result for that request, and use that same instance everywhere it's needed. You never get inconsistent state from one dependency being called twice with different results.

```
Request arrives
      │
      ▼
FastAPI builds the dependency graph:

list_matches
  └── get_query_bundle
        ├── MatchFilters       ← called once, result cached
        ├── DateRangeFilters   ← called once, result cached
        ├── PaginationParams   ← called once, result cached
        └── SortParams         ← called once, result cached

FastAPI resolves leaves first (bottom up),
then assembles the result upward.
```

This caching behaviour is per-request — the next request starts fresh.

```python title="app/routes/matches.py"
    pagination: Annotated[PaginationParams, Depends()],
    sort_params: Annotated[SortParams, Depends()],
) -> QueryBundle:
    return QueryBundle(
        filters=filters,
        date_range=date_range,
        pagination=pagination,
        sort_params=sort_params,
    )
```

```python title="app/routes/matches.py"
def list_matches(q: Annotated[QueryBundle, Depends(get_query_bundle)]):  # !mark
    results = INITIAL_DATA

        results = [m for m in results if m.sport == q.filters.sport]  # !mark

    results = results[q.pagination.offset : q.pagination.offset + q.pagination.limit]  # !mark

    }
```

The URL still works identically — `GET /matches?sport=football&page=2` — because FastAPI still reads the query string into `MatchFilters` and `PaginationParams` via their dependencies. The only change is how those values are delivered to your function.

## What You've Learned

- **Classes as dependencies** — `Depends()` calls a class's `__init__`, creating an instance from request data; Pydantic models work perfectly for this
- **`Depends()` vs `Depends(SomeClass)`** — they're equivalent when the annotation type matches; FastAPI infers the class from the type hint
- **Sub-dependencies** — a dependency function whose parameters are themselves dependencies; FastAPI builds a graph and resolves everything in order
- **`arbitrary_types_allowed`** — Pydantic config needed when a model contains fields that are already-validated Pydantic model instances
- **Per-request caching** — FastAPI calls each dependency once per request, even if it's referenced multiple times in the graph
- **`QueryBundle`** — a container model that groups all query-param dependencies, giving `list_matches` a single clean parameter instead of four
