## What is a Dependency?

A dependency is just a function. You write it like any other Python function. The special part is that you tell FastAPI "before you run my route function, run *this* function first, and give me whatever it returns."

Here's the simplest possible example — not from the project, just to show the shape:

```python
def get_current_user():
    return {"name": "Alice", "role": "admin"}

@router.get("/dashboard")
def dashboard(user: Annotated[dict, Depends(get_current_user)]):
    return {"message": f"Welcome, {user['name']}"}
```

When a request hits `GET /dashboard`, FastAPI:
1. Sees that `dashboard` needs `user`, which comes from `Depends(get_current_user)`
2. Calls `get_current_user()` automatically
3. Takes whatever it returned (`{"name": "Alice", "role": "admin"}`)
4. Passes it into `dashboard` as the `user` argument
5. Runs `dashboard` with that value in hand

Your route function never calls `get_current_user()` directly. FastAPI does it for you. That's what "injection" means — the dependency's return value is *injected* into your function.

<Callout type="info" title="Why 'dependency injection'?">
  The term sounds complicated but the idea is simple. Normally your function is responsible for getting everything it needs — you'd call `get_current_user()` yourself inside the function body. With dependency injection, you *declare* what you need and something else (FastAPI) is responsible for providing it.

  The benefit is that the thing providing the value can be swapped out — in tests, you can replace `get_current_user` with a mock that returns a fake user, without changing any route code. You'll see this payoff clearly when you write tests later.
</Callout>

## Dependencies Can Use Path Parameters Too

A dependency function can declare the same parameters as a route function — path parameters, query parameters, anything FastAPI knows how to extract from a request.

This is the key insight for the `get_match_or_404` refactor. The dependency can declare `match_id: int` as a path parameter and FastAPI will extract it from the URL automatically, just like it would for a route function.

```python
def get_match_or_404(match_id: Annotated[int, Path(ge=1, title="Match ID")]) -> Match:
    for match in INITIAL_DATA:
        if match.id == match_id:
            return match
    raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail="Match not found")
```

This is a normal Python function. It takes `match_id` from the URL path, loops through the data, returns the match if found, raises `HTTPException` if not. The `-> Match` return type annotation is just good practice — it tells Python (and you) what this function gives back.

FastAPI will call this function before any route that depends on it, passing in the `match_id` from the URL automatically.

## Step 2 — Simplify `get_match`

Before:

```python title="Before"
@router.get("/{match_id}", response_model=MatchResponse, response_model_exclude_none=True)
```

The route function went from 5 lines of logic to 2. FastAPI calls `get_match_or_404`, which looks up the match (using the `{match_id}` from the URL), and hands the result to `get_match` as `match`. If the match doesn't exist, `get_match_or_404` raises the 404 before `get_match` even runs.

Notice the parameter name changed from `match_id: int` to `match: Match`. The route no longer cares about the ID — it just wants the match object. The dependency handles the ID extraction.

## Step 3 — Simplify `update_match`

Before, `update_match` had to find the match, get its index, and then do the update logic — all mixed together:

```python title="Before"

            stored_match_data = match.model_dump()
            update_data = update.model_dump(exclude_unset=True)
            ...
    raise HTTPException(...)  # 404
```

After, the route receives the already-found `match` from the dependency. To get its index for the in-place replacement, use `INITIAL_DATA.index(match)`:

```python title="After"
def update_match(
    match: Annotated[Match, Depends(get_match_or_404)],  # !mark
    update: Annotated[MatchUpdate, Body(...)],
):
    i = INITIAL_DATA.index(match)  # !mark
    stored_match_data = match.model_dump()
    update_data = update.model_dump(exclude_unset=True)
    updated_match_dict = {**stored_match_data, **update_data}

    try:
        validated_match = Match.model_validate(updated_match_dict)
    except ValueError as e:
        raise HTTPException(
            status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
            detail=str(e),
        )

    INITIAL_DATA[i] = validated_match
    return validated_match
```

`INITIAL_DATA.index(match)` returns the position of the match object in the list. This works because the dependency returns the exact same object that lives in `INITIAL_DATA`, so Python's identity check finds it immediately.

The 404 logic is completely gone from the function body. The update logic is now clearly the only thing this function does.

## Step 4 — Simplify `replace_match`

Before, there was a naming conflict — the route had a `match` parameter for the *existing* match (found by ID) and another `match` parameter for the *new* match from the request body. Mixing those up was easy.

After splitting them with the dependency:

```python title="After"
def replace_match(
    match: Annotated[Match, Depends(get_match_or_404)],  # !mark — the existing match
    new_match: Annotated[Match, Body(openapi_examples={...})],  # !mark — the replacement
):
    i = INITIAL_DATA.index(match)
    new_match.id = match.id  # preserve the original ID
    INITIAL_DATA[i] = new_match
    return new_match
```

The names are now unambiguous: `match` is what's already stored, `new_match` is what the client sent. No more `existing_match` vs `match` confusion.

## How FastAPI Resolves Dependencies

It's worth understanding what FastAPI actually does when it sees `Depends(get_match_or_404)`:

```
Request arrives: GET /matches/3
                       │
                       ▼
        FastAPI reads the route signature:
        def get_match(match: Annotated[Match, Depends(get_match_or_404)])
                       │
                       ▼
        FastAPI sees Depends(get_match_or_404)
        → reads get_match_or_404's signature
        → sees match_id: Annotated[int, Path(ge=1)]
        → extracts "3" from the URL path
        → calls get_match_or_404(match_id=3)
                       │
                  ┌────┴────┐
                  │         │
              found       not found
                  │         │
          returns Match   raises HTTPException (404)
                  │
                  ▼
        FastAPI injects result into get_match as "match"
                  │
                  ▼
        get_match runs: return match
```

The route function itself never touches the URL. It just receives the already-validated, already-looked-up `Match` object. If anything in the dependency fails — 404, validation error, anything — the route function never runs at all.

<Callout type="info" title="The return type annotation matters">
  Notice `-> Match` on `get_match_or_404`. FastAPI uses this to understand what type the dependency returns, which helps with type checking and IDE autocompletion. When you write `match: Annotated[Match, Depends(get_match_or_404)]`, your IDE knows `match` is a `Match` object and gives you proper autocomplete on `match.id`, `match.sport`, etc.

  You can omit it and things still work, but annotating return types is good practice and makes the code easier to read.
</Callout>

## What You've Learned

- **What a dependency is** — a function FastAPI calls before your route, injecting its return value as a parameter
- **`Depends(fn)`** — tells FastAPI to call `fn` and inject the result; used inside `Annotated`
- **Dependencies can declare path/query params** — FastAPI extracts them from the request just like it would for a route function
- **`-> ReturnType` annotation** — tells FastAPI and your IDE what type the dependency returns
- **`list.index(item)`** — finds the position of an object in a list; works because the dependency returns the same object instance that lives in the list
- **`list.remove(item)`** — removes an item by value rather than by index; clean for delete after the dependency already found the object
- **Single responsibility** — each route function now does exactly one thing; the lookup concern lives in the dependency
