So far all the dependencies you've written are injected *into* a route function as a parameter — the route receives the return value and uses it. But some things you want to run for every request don't produce a value your route needs. You want them to run for their effect: checking a header, logging the request, enforcing a rule.

Forcing those into function parameters just to run them is messy:

```python
# ❌ Awkward — you don't need api_key_result, you just want it to run
def get_match(
    match: Annotated[Match, Depends(get_match_or_404)],
    _: Annotated[None, Depends(verify_api_key)],  # unused, just to trigger it
):
    return match
```

FastAPI has a cleaner answer: register dependencies directly on a decorator, a router, or the whole app — so they run for every request to that scope without cluttering any function signature.

By the end of this chapter you'll have:
- A `verify_api_key` dependency that protects every `/matches` route
- That dependency registered at the router level — one line, covers everything
- A clear understanding of the three scopes: decorator, router, app

## The Three Scopes

Dependencies can be registered at three levels. Each level covers a wider set of routes:

| Scope | Syntax | Runs for |
|-------|--------|---------|
| Single route | `@router.get("/path", dependencies=[Depends(fn)])` | That one endpoint |
| Whole router | `APIRouter(dependencies=[Depends(fn)])` | Every route on that router |
| Whole app | `app = FastAPI(dependencies=[Depends(fn)])` | Every route in the entire API |

The key difference from parameter injection: these dependencies run but their return values are discarded. Use this scope when the dependency does its job by either succeeding silently or raising an `HTTPException` to stop the request.

## Step 1 — Write `verify_api_key`

A real API needs to control who can access it. A simple first step is requiring an API key in a request header. Clients include the header `X-API-Key: your-secret` with every request. The dependency checks the value and raises `401 Unauthorized` if it's wrong or missing.

Add this to `app/routes/matches.py`, alongside the other dependency functions:

```python title="app/routes/matches.py"
from fastapi import APIRouter, Body, Depends, Header, HTTPException, Path, status  # !mark (added Header)

def verify_api_key(x_api_key: Annotated[str, Header()]) -> None:  # !mark
    if x_api_key != "sports-api-secret":  # !mark
        raise HTTPException(  # !mark
            status_code=status.HTTP_401_UNAUTHORIZED,  # !mark
            detail="Invalid or missing API key",  # !mark
        )  # !mark
```

A few things to note:

**`Header()`** works like `Path()` and `Query()` — it's a FastAPI parameter declaration that tells FastAPI where to find the value. FastAPI automatically converts the parameter name from `x_api_key` (Python snake_case) to `X-API-Key` (HTTP header format). The conversion is: replace underscores with hyphens, capitalise each word.

**`-> None`** is the return type. This dependency doesn't return anything useful — it either passes silently or raises an exception. The `None` return type signals that clearly.

**The hardcoded secret** is fine for now since this is a learning project. In a real API, you'd load this from an environment variable (covered in a later chapter).

<Callout type="info" title="401 vs 403 — what's the difference?">
  `401 Unauthorized` means "you didn't prove who you are" — wrong or missing credentials.
  `403 Forbidden` means "I know who you are, but you're not allowed to do this" — valid credentials but insufficient permissions.

  An invalid API key is a `401` because the request hasn't authenticated itself. If you later add user roles and someone with a valid key tries to access something they're not permitted to, that's a `403`.
</Callout>

## Step 2 — Register It on the Router

Instead of adding `verify_api_key` to every single route's parameters or decorator, register it once on the router:

```python title="app/routes/matches.py"
# Before
router = APIRouter(prefix="/matches", tags=["Matches"])

# After
router = APIRouter(
    prefix="/matches",
    tags=["Matches"],
    dependencies=[Depends(verify_api_key)],  # !mark
)
```

That's it. Every route registered on this router — `GET /matches`, `GET /matches/{id}`, `POST /matches`, all of them — now requires the `X-API-Key` header. No changes to any route function needed.

The `dependencies=` parameter accepts a list, so you can pass multiple dependencies:

```python
router = APIRouter(
    prefix="/matches",
    tags=["Matches"],
    dependencies=[
        Depends(verify_api_key),
        Depends(some_other_check),
    ],
)
```

All of them run for every request, in order, before any route function.

## Decorator-Level Dependencies

Sometimes you only want a dependency to run for a specific route, not the whole router. You can add it directly to the route decorator:

```python title="Example — single route protection"
@router.delete(
    "/{match_id}",
    response_model=MatchResponse,
    response_model_exclude_none=True,
    dependencies=[Depends(verify_admin_role)],  # only DELETE needs this
)
def delete_match(match: Annotated[Match, Depends(get_match_or_404)]):
    INITIAL_DATA.remove(match)
    return match
```

`verify_admin_role` runs only for `DELETE /matches/{match_id}`, not for any other route. This is useful when different operations need different protection levels — anyone can read matches, but only admins can delete them.

<Callout type="info" title="Decorator dependencies stack with router dependencies">
  If your router has `dependencies=[Depends(verify_api_key)]` and a specific route adds `dependencies=[Depends(verify_admin_role)]`, both run for that route — the router-level one first, then the decorator-level one. They don't replace each other, they stack.
</Callout>

## App-Level Dependencies

For things that should run for every single request in your entire API — not just matches — register the dependency on the `FastAPI` app itself:

```python title="main.py — hypothetical app-level dependency"
from fastapi import FastAPI, Depends

def log_every_request(request: Request) -> None:
    print(f"{request.method} {request.url.path}")

app = FastAPI(
    title="Sports Matches API",
    dependencies=[Depends(log_every_request)],  # runs for every route
)
```

`Request` is a FastAPI/Starlette class that gives you access to the raw incoming HTTP request — its method, URL, headers, client IP, and more. You can declare it as a parameter in any dependency or route function and FastAPI injects it automatically.

App-level dependencies are appropriate for things like:
- Logging every request
- Global rate limiting
- Adding a request ID header to every response

For this project, the `/health` endpoint should remain publicly accessible without an API key — it's used by monitoring systems to check if the server is running. Registering `verify_api_key` at the router level (not the app level) keeps `/health` unauthenticated while protecting all match routes.

## The Complete Dependency Picture for a `PATCH` Request

Here's everything that happens, in order, when `PATCH /matches/3` arrives with the API key header:

```
PATCH /matches/3
X-API-Key: sports-api-secret
Body: {"venue": "Wembley Stadium"}

        │
        ▼
1. Router-level: verify_api_key(x_api_key="sports-api-secret")
   → key is valid, continues silently
        │
        ▼
2. Route parameter: get_match_or_404(match_id=3)
   → finds match with id=3 in INITIAL_DATA
   → returns Match object
        │
        ▼
3. Route parameter: MatchUpdate parsed from body
   → venue="wembley stadium" (normalised)
        │
        ▼
4. update_match(match=<Match id=3>, update=<MatchUpdate venue="wembley stadium">)
   → merges, validates, saves, returns updated match
        │
        ▼
5. response_model=MatchResponse filters output
        │
        ▼
200 OK — clean MatchResponse JSON
```

Each step either completes and passes control to the next, or raises an `HTTPException` that immediately returns an error response to the client.

## Parameter Dependencies vs Decorator Dependencies — When to Use Which

| Use case | How to register |
|----------|----------------|
| You need the return value (a user, a match, a config) | Function parameter: `Annotated[T, Depends(fn)]` |
| You need it to run but don't use the return value | Decorator/router/app: `dependencies=[Depends(fn)]` |
| Runs for one specific endpoint | Route decorator: `@router.get("/path", dependencies=[...])` |
| Runs for all routes on a resource | Router: `APIRouter(dependencies=[...])` |
| Runs for every route in the API | App: `FastAPI(dependencies=[...])` |

---

## What You've Learned

- **Decorator dependencies** — `dependencies=[Depends(fn)]` on a route, router, or app runs the dependency without injecting its return value
- **`Header()`** — like `Path()` and `Query()`, tells FastAPI to read a value from an HTTP header; underscores in the parameter name become hyphens in the header name
- **Router-level dependencies** — `APIRouter(dependencies=[...])` applies a dependency to every route on that router; the cleanest way to protect a whole resource
- **App-level dependencies** — `FastAPI(dependencies=[...])` applies to every route in the entire API
- **Stacking** — router-level and decorator-level dependencies both run; they don't replace each other
- **`Request`** — a special parameter FastAPI injects automatically; gives access to the raw HTTP request object
- **`401` vs `403`** — wrong/missing credentials vs valid credentials with insufficient permissions
