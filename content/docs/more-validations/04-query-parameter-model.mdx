# Query Parameter Models

Our `/matches` endpoint works, but the function signature is getting long. Seven query parameters make it hard to read and maintain. Let's refactor.

## The Current Problem

Here's what our endpoint looks like now:

```python title="matches.py - Current implementation"
@app.get("/matches", tags=["Matches"])
def list_matches(
    sport: Annotated[Optional[Sport], Query()] = None,
    status: Annotated[Optional[Status], Query()] = None,
    winner: Annotated[Optional[Winner], Query()] = None,
    team: Annotated[Optional[str], Query(min_length=3, max_length=50)] = None,
    match_date: Annotated[Optional[date], Query()] = None,
    date_filter: Annotated[Optional[DateFilter], Query()] = None,
    page: Annotated[int, Query(ge=1)] = 1,
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
):
    # ... 30+ lines of filtering logic ...
```

<Callout type="warn" title="Problems with This Approach">
  - **Long function signature** — 8 parameters make it hard to scan
  - **No grouping** — pagination params mixed with filter params
  - **Can't reuse** — if you add another endpoint that needs the same filters, you copy-paste everything
  - **Hard to validate relationships** — checking if `match_date` requires `date_filter` happens inside the function
</Callout>

---

## The Solution: Query Parameter Models

Group related parameters into Pydantic models. Since FastAPI 0.115.0, you can pass an entire model as a query parameter:

```python title="matches.py"
from typing import Annotated, Optional, Literal  # !mark (add Literal)
from pydantic import BaseModel, Field  # !mark (add Field)

# ... existing imports ...


class MatchFilters(BaseModel):  # !mark
    sport: Optional[Literal["football", "cricket", "hockey", "basketball", "baseball"]] = None  # !mark
    status: Optional[Literal["upcoming", "cancelled", "abandoned", "completed"]] = None  # !mark
    winner: Optional[Literal["home_team", "away_team", "draw"]] = None  # !mark
    team: Optional[str] = Field(None, min_length=3, max_length=50)  # !mark


@app.get("/matches", tags=["Matches"])
def list_matches(
    filters: Annotated[MatchFilters, Query()],  # !mark
):
    # Access filters.sport, filters.status, etc.
    matches = INITIAL_DATA
    
    if filters.sport:
        matches = [m for m in matches if m.sport.value == filters.sport]
    
    if filters.status:
        matches = [m for m in matches if m.status.value == filters.status]
    
    # ... rest of filtering logic ...
    
    return matches
```

<Callout type="info" title="What Changed?">
  1. **Created `MatchFilters` class** — groups all filter parameters together
  2. **Used `Literal` instead of `Enum`** — simpler and more explicit
  3. **Used `Field()` for validation** — same as `Query()` but for model fields
  4. **Single function parameter** — `filters: MatchFilters` instead of 4 separate params
</Callout>

## Understanding Literal

`Literal` is a Python type hint that specifies exact allowed values:

```python title="Literal vs Enum comparison"
# OLD: Enum approach
class Sport(str, Enum):
    football = "football"
    cricket = "cricket"
    basketball = "basketball"

sport: Optional[Sport] = None


# NEW: Literal approach
sport: Optional[Literal["football", "cricket", "basketball"]] = None
```

### Why Literal is Better for Query Parameters

| Feature | Enum | Literal |
|---------|------|---------|
| **Simplicity** | Requires class definition | Inline definition |
| **Reusability** | Can reuse across files | Definition tied to location |
| **Type checking** | ✅ Editor support | ✅ Editor support |
| **FastAPI docs** | ✅ Shows options | ✅ Shows options |
| **JSON serialization** | Need `(str, Enum)` | Just works |
| **Model fields** | Works but verbose | Clean and clear |

<Callout type="info" title="When to Use Each">
  - **Use Enums** in your **data models** (Pydantic classes that represent database tables or API responses) — they're reusable and self-documenting
  - **Use Literals** in your **query parameters** and function arguments — they're inline and clearer for one-off use cases
  
  For our sports matches API:
  - `Match` model still uses `Sport` enum (reused across multiple matches)
  - Query filters use `Literal` (specific to this endpoint)
</Callout>

```python title="matches.py"
class PaginationParams(BaseModel):  # !mark

@app.get("/matches", tags=["Matches"])

    filters: Annotated[MatchFilters, Query()],
    pagination: Annotated[PaginationParams, Query()],  # !mark

    matches = INITIAL_DATA
    
    # Apply filters...
    if filters.sport:
        matches = [m for m in matches if m.sport.value == filters.sport]
    
    # Apply pagination
    start = (pagination.page - 1) * pagination.limit  # !mark
```

  ```python
                     ^^  ^^^^^^^^^^^^  ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  ```

  `Field()` works exactly like `Query()`, but it's for model fields. The first positional argument is the default value, then you add constraints and metadata.

## Adding Date Range Filters

Let's improve our date filtering by creating a dedicated model:

```python title="matches.py"
class DateRangeFilters(BaseModel):  # !mark
    from_date: Optional[date] = Field(None, description="Matches on or after this date")  # !mark
    to_date: Optional[date] = Field(None, description="Matches on or before this date")  # !mark


@app.get("/matches", tags=["Matches"])
def list_matches(
    filters: Annotated[MatchFilters, Query()],
    date_range: Annotated[DateRangeFilters, Query()],  # !mark
    pagination: Annotated[PaginationParams, Query()],
):
    matches = INITIAL_DATA
    
    # Apply match filters
    if filters.sport:
        matches = [m for m in matches if m.sport.value == filters.sport]
    
    if filters.status:
        matches = [m for m in matches if m.status.value == filters.status]
    
    if filters.team:
        team_lower = filters.team.lower()
        matches = [
            m for m in matches
            if team_lower in m.home_team.lower() or team_lower in m.away_team.lower()
        ]
    
    # Apply date filters
    if date_range.from_date:  # !mark
        matches = [m for m in matches if m.date >= date_range.from_date]  # !mark
    
    if date_range.to_date:  # !mark
        matches = [m for m in matches if m.date <= date_range.to_date]  # !mark
    
    # Validate date range relationship
    if date_range.from_date and date_range.to_date:  # !mark
        if date_range.from_date > date_range.to_date:  # !mark
            return {"error": "from_date must be before or equal to to_date"}  # !mark
    
    # Apply pagination
    start = (pagination.page - 1) * pagination.limit
    end = start + pagination.limit
    
    return {
        "total": len(matches),
        "page": pagination.page,
        "limit": pagination.limit,
        "results": matches[start:end],
    }
```

<Callout type="info" title="Simpler Date Filtering">
  Instead of `match_date` + `date_filter` (with `gt`/`lt`/`eq` operators), we now have:
  - `from_date` — matches on or after this date
  - `to_date` — matches on or before this date
  
  This is more intuitive for users: `/matches?from_date=2025-01-01&to_date=2025-12-31` clearly means "matches in 2025."
</Callout>

## Model-Level Validation

You can add validation logic **inside the model** using Pydantic's `model_validator`:

```python title="matches.py"
from pydantic import model_validator  # !mark


class DateRangeFilters(BaseModel):
    from_date: Optional[date] = Field(None, description="Matches on or after this date")
    to_date: Optional[date] = Field(None, description="Matches on or before this date")
    
    @model_validator(mode='after')  # !mark
    def validate_date_range(self):  # !mark
        if self.from_date and self.to_date:  # !mark
            if self.from_date > self.to_date:  # !mark
                raise ValueError("from_date must be before or equal to to_date")  # !mark
        return self  # !mark
```

Now the date range validation happens **automatically** when the model is created:

```python title="matches.py"
@app.get("/matches", tags=["Matches"])
def list_matches(
    filters: Annotated[MatchFilters, Query()],
    date_range: Annotated[DateRangeFilters, Query()],  # !mark (validation happens here)
    pagination: Annotated[PaginationParams, Query()],
):
    # No need to check date_range.from_date > date_range.to_date manually!
    # Pydantic already did it. If invalid, FastAPI returns error before this runs.
    
    matches = INITIAL_DATA
    # ... filtering logic ...
```

<Callout type="info" title="@model_validator Explained">
  - `mode='after'` means "run this after Pydantic has parsed all fields"
  - The function receives `self` — the model instance with all fields populated
  - Raise `ValueError` to reject invalid combinations
  - Return `self` if everything is valid
  
  This moves validation **into the model definition**, making it automatic and reusable.
</Callout>

```python title="matches.py"
from pydantic import BaseModel, Field, model_validator
```

## Forbidding Extra Parameters

By default, Pydantic ignores extra parameters. If a user sends `/matches?sport=football&random_field=value`, the `random_field` is silently ignored.

Sometimes you want to be strict and reject unknown parameters:

```python title="matches.py"
class MatchFilters(BaseModel):
    model_config = {"extra": "forbid"}  # !mark
    
    sport: Optional[Literal["football", "cricket", "hockey", "basketball", "baseball"]] = None
    status: Optional[Literal["upcoming", "cancelled", "abandoned", "completed"]] = None
    winner: Optional[Literal["home_team", "away_team", "draw"]] = None
    team: Optional[str] = Field(None, min_length=3, max_length=50)
```

Now try:

```bash
curl "http://localhost:8000/matches?sport=football&typo_field=oops"
```

Response:

```json title="Validation error"
{
  "detail": [
    {
      "type": "extra_forbidden",
      "loc": ["query", "typo_field"],
      "msg": "Extra inputs are not permitted",
      "input": "oops"
    }
  ]
}
```

<Callout type="warn" title="When to Use extra='forbid'">
  **Use it for:**
  - Internal APIs where you control all clients
  - APIs where typos should be caught immediately
  - Security-sensitive endpoints
  
  **Don't use it for:**
  - Public APIs (breaks forward compatibility)
  - APIs consumed by many different clients
  - APIs where you might add parameters later
  
  The default (ignore extra params) is usually the right choice for public APIs.
</Callout>

## Mixing Enum and Literal

You can still use Enums in your **data models** while using Literals in **query parameters**:

```python title="matches.py"
from enum import Enum

# Data model uses Enum (reusable across app)
class Sport(str, Enum):
    football = "football"
    cricket = "cricket"
    basketball = "basketball"
    baseball = "baseball"
    hockey = "hockey"


class Match(BaseModel):
    id: int
    sport: Sport  # !mark (Enum in data model)
    # ... other fields ...


# Query parameter uses Literal (specific to this endpoint)
class MatchFilters(BaseModel):
    sport: Optional[Literal["football", "cricket", "basketball", "baseball", "hockey"]] = None  # !mark


@app.get("/matches")
def list_matches(filters: Annotated[MatchFilters, Query()]):
    matches = INITIAL_DATA
    
    if filters.sport:
        # Compare Literal (string) with Enum (needs .value)
        matches = [m for m in matches if m.sport.value == filters.sport]  # !mark
    
    return matches
```

<Callout type="info" title="Why Keep Enums in Data Models?">
  Enums in data models provide:
  - Type safety when creating matches: `Match(sport=Sport.football, ...)`
  - IDE autocomplete: `match.sport.` shows all options
  - Centralized definition: Change sport list once, all models update
  
  Literals in queries provide:
  - Simpler syntax for API consumers
  - No need to import Enum in client code
  - Direct string comparison
</Callout>

## When to Use Query Parameter Models

Use models when:

✅ You have **3+ related parameters** (filters, pagination, sorting)  
✅ You want to **reuse the same parameters** across multiple endpoints  
✅ You need **cross-field validation** (like date range checks)  
✅ You want **better documentation** with grouped parameters  

Don't use models when:

❌ You only have **1-2 simple parameters**  
❌ Parameters are **completely unrelated** (no logical grouping)  
❌ The endpoint is **one-off and never reused**

## Summary: Model-Based Query Parameters

### Old Approach (Individual Parameters)

```python

    sport: Annotated[Optional[Sport], Query()] = None,
    status: Annotated[Optional[Status], Query()] = None,
    team: Annotated[Optional[str], Query(min_length=3)] = None,
    # ... 5 more parameters ...
):
    if from_date and to_date and from_date > to_date:
        return {"error": "Invalid date range"}
    # ...
```

**Problems:** Long signature, repeated validation, hard to reuse

### New Approach (Model Parameters)

```python
class MatchFilters(BaseModel):
    sport: Optional[Literal["football", "cricket", "basketball", "baseball", "hockey"]] = None
    status: Optional[Literal["upcoming", "cancelled", "abandoned", "completed"]] = None
    team: Optional[str] = Field(None, min_length=3, max_length=50)


class DateRangeFilters(BaseModel):
    from_date: Optional[date] = None
    to_date: Optional[date] = None
    
    @model_validator(mode='after')
    def validate_date_range(self):
        if self.from_date and self.to_date and self.from_date > self.to_date:
            raise ValueError("from_date must be before or equal to to_date")
        return self
```
