# Updating Resources (PUT & PATCH)

You've learned to create matches with `POST` and retrieve them with `GET`. Now it's time to update existing resources. HTTP provides two methods for updates: `PUT` for full replacement and `PATCH` for partial updates. In this lesson, you'll learn when to use each and how to implement them with proper validation.

---

## Where We Are

Your API currently supports:
- `POST /matches` — Create new matches
- `GET /matches` — List matches with filters
- `GET /matches/{match_id}` — Get a specific match

But what happens when:
- A match venue changes?
- You need to record the final score?
- The match date gets rescheduled?
- You want to add lineups after the match is created?

You need update endpoints.

---

## PUT vs PATCH: The Difference

HTTP defines two methods for updates, and they're not interchangeable:

| | PUT | PATCH |
|---|-----|-------|
| **Meaning** | Replace entire resource | Update specific fields |
| **Requires** | All fields (complete replacement) | Only fields being changed |
| **Idempotent** | Yes — same request = same result | Yes — same request = same result |
| **Use when** | You have complete new data | You want to change a few fields |

**Example scenario:**

```python
# Current match
{
  "id": 1,
  "home_team": "Arsenal",
  "away_team": "Chelsea",
  "sport": "football",
  "date": "2026-03-15",
  "status": "upcoming",
  "venue": "Emirates Stadium"
}
```

**With PUT** — Must send all fields:
```json
PUT /matches/1
{
  "home_team": "Arsenal",
  "away_team": "Chelsea",
  "sport": "football",
  "date": "2026-03-20",     ← Changed
  "status": "upcoming",
  "venue": "Emirates Stadium"
}
```

**With PATCH** — Send only what changed:
```json
PATCH /matches/1
{
  "date": "2026-03-20"  ← Only this
}
```

<Callout type="info" title="When to Use Which">
- **Use PATCH** for most updates — it's more flexible and efficient
- **Use PUT** when you truly need to replace the entire resource
- In practice, APIs often provide PATCH but not PUT, since partial updates are more common
</Callout>

## Step 1 — Create a PATCH Endpoint

Start with PATCH since it's the most commonly needed. First, create a model for updates:

```python title="matches.py - add before endpoints"
class MatchUpdate(BaseModel):
    """Model for partial updates to a match. All fields are optional."""
    venue: Optional[str] = Field(
        None,
        min_length=3,
        max_length=100,
        examples=["Wembley Stadium"],
    )
    date: Optional[date] = Field(None, examples=["2026-04-20"])
    status: Optional[Status] = None
    winner: Optional[Winner] = None
    score_home: Optional[int] = Field(None, ge=0, examples=[2])
    score_away: Optional[int] = Field(None, ge=0, examples=[2])
    home_lineup: Optional[List[Player]] = None
    away_lineup: Optional[List[Player]] = None
    events: Optional[List[MatchEvent]] = None
    statistics: Optional[MatchStatistics] = None
```

Notice every field is `Optional` and has a default of `None`. This is the key difference from `Match`:

```python title="Comparing Match vs MatchUpdate"
# Match — some fields required
class Match(BaseModel):
    home_team: str  # Required!
    date: date      # Required!
    
# MatchUpdate — everything optional
class MatchUpdate(BaseModel):
    venue: Optional[str] = None  # Optional
    date: Optional[date] = None   # Optional
```

Why? Because with PATCH, the user only sends fields they want to change. If they don't send `venue`, it means "don't change the venue," not "set venue to None."

Now add the endpoint:

```python title="matches.py"
@app.patch("/matches/{match_id}", tags=["Matches"])

    match_id: Annotated[int, Path(ge=1, title="Match ID")],

    Partially update a match (PATCH).
    
    Only provided fields will be updated. Omitted fields remain unchanged.
    """

        if match.id == match_id:
            # Update only the provided fields
            update_data = update.model_dump(exclude_unset=True)
            
            for field, value in update_data.items():
                setattr(match, field, value)

```

Breaking down the key line:

```python title="Understanding exclude_unset"
update_data = update.model_dump(exclude_unset=True)
#                               ^^^^^^^^^^^^^^^^^^
#                               Only include fields the user actually sent
```

`exclude_unset=True` means "give me a dict with only the fields that were explicitly set by the user, not fields that defaulted to None."

Example:
```python
# User sends: {"venue": "Wembley Stadium"}
update_data = {"venue": "Wembley Stadium"}  # Only this field

# Without exclude_unset=True, you'd get:
update_data = {
    "venue": "Wembley Stadium",
    "date": None,
    "status": None,
    "winner": None,
    # ... all fields, even ones not sent
}
```

<Callout type="warn" title="Why exclude_unset Matters">
Without `exclude_unset=True`, fields the user didn't send would be set to `None`, potentially erasing existing data. You'd turn a venue update into accidentally clearing the match date, status, and scores!
</Callout>

## Step 2 — Add Validation to Updates

The current implementation has a problem: you can create invalid states by updating fields in isolation:

```json title="This should fail but doesn't"
PATCH /matches/4
{
  "status": "completed"
}
```

You just marked a match as completed without providing scores or a winner. The match is now in an invalid state!

Fix this by adding validation to `MatchUpdate`:

```python title="matches.py"
class MatchUpdate(BaseModel):
    """Model for partial updates to a match. All fields are optional."""
    venue: Optional[str] = Field(None, min_length=3, max_length=100)
    date: Optional[date] = None
    status: Optional[Status] = None
    winner: Optional[Winner] = None
    score_home: Optional[int] = Field(None, ge=0)
    score_away: Optional[int] = Field(None, ge=0)
    home_lineup: Optional[List[Player]] = None
    away_lineup: Optional[List[Player]] = None
    events: Optional[List[MatchEvent]] = None
    statistics: Optional[MatchStatistics] = None

    @model_validator(mode='after')  # !mark
    def validate_update_logic(self):  # !mark
        # If updating to completed, scores must be provided
        if self.status == Status.completed:  # !mark
            if self.score_home is None or self.score_away is None:  # !mark
                raise ValueError(  # !mark
                    "Cannot set status to completed without providing both scores"  # !mark
                )  # !mark
        return self  # !mark
```

Now try that invalid update again:

```json title="Request"
{
  "status": "completed"
}
```

Response:
```json title="Validation error"
{
  "detail": [
    {
      "loc": ["body"],
      "msg": "Cannot set status to completed without providing both scores",
      "type": "value_error"
    }
  ]
}
```

The validator catches the problem before the update is applied.

But there's still an issue: what if the existing match already has `score_home` but you're only updating `score_away`? The validator only sees the update fields, not the complete match.

<Callout type="warn" title="The Partial Update Validation Problem">
`MatchUpdate` validators can only validate the fields being updated, not the complete final state. You need a second layer of validation that checks the complete match after the update is applied.
</Callout>

---

## Step 3 — Validate the Complete Match After Updates

Update your endpoint to re-validate the complete match after applying changes:

```python title="matches.py"
@app.patch("/matches/{match_id}", tags=["Matches"])
def update_match(
    match_id: Annotated[int, Path(ge=1, title="Match ID")],
    update: MatchUpdate,
):
    """Partially update a match with validation of the final state."""
    for match in INITIAL_DATA:
        if match.id == match_id:
            # Update only the provided fields
            update_data = update.model_dump(exclude_unset=True)
            
            for field, value in update_data.items():
                setattr(match, field, value)
            
            # Re-validate the complete match after updates  # !mark
            try:  # !mark
                Match.model_validate(match)  # !mark
            except ValueError as e:  # !mark
                return {"error": f"Update would create invalid match: {str(e)}"}, 400  # !mark
            
            return match
    
    return {"error": "Match not found"}, 404
```

`Match.model_validate(match)` validates the complete match object using all the validation rules from the `Match` model. If the update would create an invalid state (like completed status without scores), it's caught here.

Now the validation happens in two phases:
1. **MatchUpdate validators** — Check that the update fields themselves are valid
2. **Match validators** — Check that the complete match (existing + updates) is valid

<Callout type="info" title="Two-Layer Validation for Updates">
- **Layer 1** (`MatchUpdate` validators): Validate the update request is reasonable
- **Layer 2** (`Match.model_validate`): Validate the final result is valid
  
This pattern ensures you can't create invalid states through partial updates.
</Callout>

## Step 4 — Add a PUT Endpoint (Full Replacement)

PUT replaces the entire resource. The model is just `Match`, not `MatchUpdate`, because all fields are required:

```python title="matches.py"
@app.put("/matches/{match_id}", tags=["Matches"])
def replace_match(
    match_id: Annotated[int, Path(ge=1, title="Match ID")],
    match: Match,
):
    """
    Replace an entire match with new data (PUT).
    
    All fields must be provided. This completely replaces the existing match.
    Use PATCH if you only want to update specific fields.
    """
    for i, existing_match in enumerate(INITIAL_DATA):
        if existing_match.id == match_id:
            match.id = match_id  # Preserve the ID
            INITIAL_DATA[i] = match
            return match
    
    return {"error": "Match not found"}, 404
```

Notice:
- Parameter is `match: Match`, not `match: MatchUpdate`
- All fields must be provided (they're required in the `Match` model)
- We preserve the ID from the URL (don't let the user change it)
- We replace the entire match in the list

## PUT vs PATCH: When to Use Each

**Use PATCH when:**
- ✅ Updating specific fields (most common case)
- ✅ User doesn't have/need the complete resource
- ✅ You want to avoid accidentally clearing unrelated fields
- ✅ Building user-friendly APIs

**Use PUT when:**
- ✅ You truly want to replace the entire resource
- ✅ Your API design requires full replacements
- ✅ You have a "reset to default" operation

**In practice:** Most modern APIs only implement PATCH. PUT is less common because full replacement is rarely what users actually need.

<Callout type="info" title="REST Purists vs Pragmatists">
REST purists say you should implement both PUT and PATCH. Pragmatists say most APIs only need PATCH since partial updates are what users actually want. This tutorial shows you both so you can make an informed choice, but in real projects, PATCH alone is often sufficient.
</Callout>

## Common Patterns for Update Validation

```python title="Validation patterns for updates"
class ResourceUpdate(BaseModel):
    # Pattern 1: Field-level validation still applies
    name: Optional[str] = Field(None, min_length=2, max_length=50)
    
    # Pattern 2: Validate update fields make sense together
    @model_validator(mode='after')
    def validate_update_fields(self):
        if self.status == "completed" and self.score is None:
            raise ValueError("Need score when marking as completed")
        return self

# Pattern 3: Validate final state after applying update
@app.patch("/resource/{id}")
def update_resource(id: int, update: ResourceUpdate):
    # Apply updates
    for field, value in update.model_dump(exclude_unset=True).items():
        setattr(resource, field, value)
    
    # Validate complete resource
    try:
        Resource.model_validate(resource)
    except ValueError as e:
        return {"error": str(e)}, 400
    
    return resource
```

This three-layer approach ensures:
1. Individual update fields are valid
2. Update fields work together
3. Final resource state is valid
