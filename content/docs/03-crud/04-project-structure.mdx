Your `matches.py` file is now over 600 lines long and doing everything at once — it defines enums, Pydantic models, seed data, and all the route handlers. This is fine when you're learning, but in any real project it becomes a problem fast. Finding a bug in a validator means scrolling past hundreds of lines of route code. Adding a new model means hunting for where the other models are. A new developer looking at the project has no idea where to start.

The fix is **project structure** — splitting the code across focused files, each with a single clear responsibility. This is not about making the code more complex. It's about making it easier to read, easier to change, and easier to grow.

---

## The Structure We're Building

Here's what the project will look like after this chapter:

```
sports_api/
├── main.py              ← creates the app, registers routes
├── app/
│   ├── __init__.py      ← marks app/ as a Python package
│   ├── models.py        ← all Pydantic models and enums
│   ├── database.py      ← the in-memory data store
│   └── routes/
│       ├── __init__.py  ← marks routes/ as a Python package
│       └── matches.py   ← all /matches route handlers
```

Four files, each with one job:

| File | Responsibility |
|------|---------------|
| `main.py` | Boot the app. That's it. |
| `app/models.py` | Every class: enums, Pydantic models, filter/param models |
| `app/database.py` | The seed data list |
| `app/routes/matches.py` | All route handlers for `/matches` |

<Callout type="info" title="What is __init__.py?">
  When Python sees a folder containing a file called `__init__.py`, it treats that folder as a **package** — meaning you can import from it using dot notation like `from app.models import Match`.

  The `__init__.py` files in this project are empty. They don't need any content; their presence alone is what tells Python "this folder is importable." You'll need to create them but you don't need to put anything inside them.
</Callout>

---

## Why Split It This Way?

Before writing any code, it's worth understanding the reasoning — because "just split it into files" isn't a rule, it's a decision you should be able to justify.

**`models.py` is separate because models are shared everywhere.** Routes import models. The database uses models. In the future you might add tests that import models, or a serialization layer. If models lived inside `routes/matches.py`, every other file that needed them would have to import from a routes file — which is backwards. Models are the foundation; they should have no dependencies on routes.

**`database.py` is separate because data and behaviour are different things.** Right now `INITIAL_DATA` is a list in memory. Later you'll replace it with a real database connection. If the data lives in its own file, you only need to change that one file — no routes, no models, nothing else changes.

**`routes/matches.py` uses a folder** because APIs grow. Today you have one resource: matches. Tomorrow you might add players, teams, stadiums. Each gets its own file: `routes/players.py`, `routes/teams.py`. If routes lived at the top level, you'd end up with a flat pile of files and no clear organisation.

**`main.py` stays thin** because it's the entry point — the first file anyone reads when they want to understand the project. A thin `main.py` that just creates the app and registers routes tells you everything you need to know in five lines.

## Step 2 — `app/models.py`

This file gets every enum, every Pydantic class, every filter/param model. Nothing else — no data, no routes, no app creation.

The imports are slightly trimmed from the original: `FastAPI`, `HTTPException`, `status`, `Body`, `Path`, `Depends` are all route/app concerns and don't belong here. `models.py` only needs `pydantic` and standard library types.

            raise ValueError("home_team and away_team must be different")

            if self.winner is None:
                raise ValueError("Completed matches must have a winner")
        if self.status != Status.completed and self.winner is not None:
            raise ValueError("Only completed matches can have a winner")
        return self


class MatchUpdate(BaseModel):
    """Model for partial updates to a match. All fields are optional."""

    venue: Optional[str] = Field(None, min_length=3, max_length=100, examples=["Wembley Stadium"])
    date: Optional[dt_date] = None
    status: Optional[Status] = Field(None, examples=["completed"])
    winner: Optional[Winner] = Field(None, examples=["home_team"])
    home_lineup: Optional[List[Player]] = None
    away_lineup: Optional[List[Player]] = None

    @field_validator("venue")
    @classmethod
    def normalize_venue(cls, value):
        if value is not None:
            return value.strip().lower()
        return value

    @model_validator(mode="after")
    def validate_update_logic(self):
        if self.status == Status.completed and self.winner is None:
            raise ValueError("Cannot set status to completed without providing a winner")
        if self.status != Status.completed and self.winner is not None:
            raise ValueError("Cannot set winner for non-completed matches")
        return self


class MatchResponse(BaseModel):
    id: int
    home_team: str
    away_team: str
    venue: Optional[str] = None
    date: dt_date
    sport: Sport
    status: Status
    winner: Optional[Winner] = None
    home_lineup: Optional[List[Player]] = None
    away_lineup: Optional[List[Player]] = None


class MatchListResponse(BaseModel):
    total: int
    page: int
    limit: int
    matches: List[MatchResponse]


class MatchFilters(BaseModel):
    sport: Optional[Sport] = Field(None, title="Sport Type", description="Filter matches by sport")
    status: Optional[Status] = Field(None, title="Match Status", description="Filter by match status")
    winner: Optional[Winner] = Field(None, title="Winner", description="Filter by match winner")
    team: Optional[str] = Field(
        None,
        min_length=3, max_length=50,
        pattern="^[A-Za-z0-9 ]+$",
        title="Team Name",
        description="Filter by team name. Case-insensitive partial match. Letters, numbers and spaces only.",
        examples=["Arsenal"],
    )
    team_filter: Optional[TeamFilter] = Field(
        None,
        title="Team Result Filter",
        description="Filter by how the team performed — won, lost, or draw. Requires team to be set.",
    )

    @model_validator(mode="after")
    def validate_winner_and_team_filter(self):
        if self.team_filter and self.winner:
            raise ValueError("team_filter and winner cannot be used together")
        if (self.team_filter or self.winner) and self.status not in (Status.completed, None):
            raise ValueError("team_filter and winner can only be used when status is 'completed'")
        return self

    @field_validator("team")
    @classmethod
    def normalize_team(cls, value):
        if value is not None:
            return value.strip().lower()
        return value


class DateRangeFilters(BaseModel):
    from_date: Optional[dt_date] = Field(
        None, title="Start Date",
        description="Filter matches on or after this date (YYYY-MM-DD)",
        examples=["2026-01-01"],
    )
    to_date: Optional[dt_date] = Field(
        None, title="End Date",
        description="Filter matches on or before this date (YYYY-MM-DD)",
        examples=["2026-12-31"],
    )

    @model_validator(mode="after")
    def validate_date_range(self):
        if self.from_date and self.to_date:
            if self.from_date > self.to_date:
                raise ValueError("from_date must be before or equal to to_date")
        return self


class PaginationParams(BaseModel):
    page: int = Field(1, ge=1, description="Page number (1-indexed)")
    limit: int = Field(20, ge=1, le=100, description="Items per page (max 100)")

    @computed_field
    @property
    def offset(self) -> int:
        return (self.page - 1) * self.limit


class SortParams(BaseModel):
    sort_by: Optional[Literal["date", "sport", "status"]] = Field(
        None, title="Sort By", description="Field to sort results by"
    )
    sort_order: Literal["asc", "desc"] = Field(
        "asc", title="Sort Order",
        description="Sort direction — asc for ascending, desc for descending",
    )

## Step 4 — `app/routes/matches.py`

This is where all the route handlers live. The key change here is that we replace `@app.get(...)` and `@app.post(...)` with an **`APIRouter`**.

### What is `APIRouter`?

`APIRouter` is FastAPI's way of grouping related routes. Think of it as a mini-app — it has its own `get`, `post`, `patch`, `put`, `delete` decorators, just like the main `app`. The difference is that you register it onto the main `app` later in `main.py`.

```python
# Instead of this:
@app.get("/matches/{match_id}", ...)
def get_match(...):

# You write this:
router = APIRouter(prefix="/matches", tags=["Matches"])

@router.get("/{match_id}", ...)
def get_match(...):
```

The `prefix="/matches"` on the router means you don't need to repeat `/matches` in every route decorator. `@router.get("/{match_id}")` becomes `GET /matches/{match_id}` when the router is registered. `@router.get("")` becomes `GET /matches`.

```python title="app/routes/matches.py"
from datetime import date as dt_date
from typing import Annotated, List

from fastapi import APIRouter, Body, Depends, HTTPException, Path, status

from app.database import INITIAL_DATA
from app.models import (
    DateRangeFilters,
    Match,
    MatchFilters,
    MatchListResponse,
    MatchResponse,
    MatchUpdate,
    PaginationParams,
    SortParams,
    Status,
    TeamFilter,
    Winner,
)

router = APIRouter(prefix="/matches", tags=["Matches"])


@router.get("", response_model=MatchListResponse, response_model_exclude_none=True)
def list_matches(
    filters: Annotated[MatchFilters, Depends()],
    date_range: Annotated[DateRangeFilters, Depends()],
    pagination: Annotated[PaginationParams, Depends()],
    sort_params: Annotated[SortParams, Depends()],
):
    # ... same implementation as before


@router.get("/{match_id}", response_model=MatchResponse, response_model_exclude_none=True)
def get_match(match_id: Annotated[int, Path(ge=1, title="Match ID")]):
    # ... same implementation


@router.post("", response_model=MatchResponse, response_model_exclude_none=True, status_code=status.HTTP_201_CREATED)
def create_match(match: Annotated[Match, Body(openapi_examples={...})]):
    # ... same implementation


@router.patch("/{match_id}", response_model=MatchResponse, response_model_exclude_none=True)
def update_match(...):
    # ... same implementation


@router.put("/{match_id}", response_model=MatchResponse, response_model_exclude_none=True)
def replace_match(...):
    # ... same implementation


@router.delete("/{match_id}", response_model=MatchResponse, response_model_exclude_none=True)
def delete_match(match_id: Annotated[int, Path(ge=1, title="Match ID")]):
    # ... same implementation
```

The function bodies are **identical** to the previous chapter. Nothing about the logic changes — only the decorator (`@router` instead of `@app`) and the path (no `/matches` prefix needed since the router has it).

## Step 5 — `main.py`

This is the entry point. It creates the `FastAPI` app, imports the router, and registers it with `app.include_router()`. That's the only new concept here.

```python title="main.py"
from fastapi import FastAPI

from app.routes.matches import router as matches_router

app = FastAPI(title="Sports Matches API")

app.include_router(matches_router)  # !mark


@app.get("/health", tags=["System"])
def health_check():
    return {"status": "ok"}
```

`app.include_router(matches_router)` takes all the routes defined on `matches_router` and registers them onto the main app, with the prefix `/matches` already baked in from when the router was created.

The `health_check` route stays in `main.py` because it's not a matches route — it's a system-level endpoint. If you ever add more system routes (like `/version` or `/metrics`), they'd live here or in a `routes/system.py` file.

### The import alias `as matches_router`

The `router` variable in `routes/matches.py` has a generic name. When you import it in `main.py`, renaming it to `matches_router` makes the code self-documenting:

```python
# ❌ What is "router"? Could be anything.
from app.routes.matches import router
app.include_router(router)

# ✅ Clear at a glance
from app.routes.matches import router as matches_router
app.include_router(matches_router)
```

When you add a second resource later, this pattern scales naturally:

```python
from app.routes.matches import router as matches_router
from app.routes.players import router as players_router

app.include_router(matches_router)
app.include_router(players_router)
```

## How Python Finds Your Files — Understanding Imports

This is the concept that trips beginners up the most when splitting a project. When you write:

```python
from app.models import Match
```

Python needs to know where `app` is. It looks for a folder called `app` in the same directory as the file doing the import, or in `sys.path` (the list of directories Python searches). This is why you run your server from the `sports_api/` root:

```bash
# Run from inside sports_api/
uvicorn main:app --reload
```

If you ran from inside `sports_api/app/`, Python wouldn't be able to find `app.models` because there's no folder called `app` relative to that location.

The `__init__.py` files are what make `app/` and `app/routes/` **importable packages**. Without them Python just sees folders, not modules.

```
sports_api/         ← run uvicorn from here
├── main.py
└── app/
    ├── __init__.py     ← makes app/ importable as "app"
    ├── models.py       ← importable as "app.models"
    ├── database.py     ← importable as "app.database"
    └── routes/
        ├── __init__.py ← makes routes/ importable as "app.routes"
        └── matches.py  ← importable as "app.routes.matches"
```

---

## The Dependency Flow

With the project split up, here's how the files depend on each other. An arrow means "imports from":

```
main.py
  └── app.routes.matches
        ├── app.models
        └── app.database
              └── app.models
```

Notice that `app.models` is at the bottom — it imports nothing from your own project, only from the standard library and Pydantic. This is the right shape: models are the foundation, everything else builds on top of them.

`main.py` is at the top — it imports everything it needs but nothing imports from it. This is also correct: the entry point should be a consumer, not a provider.

<Callout type="warn" title="Never import from main.py">
  If another file ever needs to import something from `main.py` (like `from main import app`), that's a sign you've put the wrong thing there. The `app` object, models, data, and routes should all be importable from their own files without touching `main.py`. Tests, in particular, will need to import `app` to spin up a test client — which is fine and works because `main.py` imports from `app/`, not the other way around.
</Callout>

## What You've Learned

- **Project structure** — splitting one large file into focused files, each with a single responsibility
- **`__init__.py`** — empty files that tell Python a folder is an importable package
- **`APIRouter`** — groups related routes with a shared prefix and tags; registered onto the main app with `include_router()`
- **`prefix=` on `APIRouter`** — removes the need to repeat the base path on every route decorator
- **Import aliases (`as`)** — renaming imports for clarity when multiple routers exist
- **The dependency flow** — models → database → routes → main; the foundation never imports from the top
- **Where to run `uvicorn`** — from the project root, so Python can resolve `app.*` imports
