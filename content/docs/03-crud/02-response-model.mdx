So far your API works — it creates, reads, updates, and deletes matches. But there's something important missing: you have no control over what your endpoints actually send back to the client.

Right now if you `return match`, FastAPI serializes the entire `Match` object, including every field, `None` values and all. If your `Match` model ever gains an internal field — like a database row version number, an internal flag, or a raw password hash — it would leak straight into the API response. The docs also show an inconsistent schema because FastAPI is guessing the return shape.

`response_model` fixes all of this. It lets you declare a separate output model and FastAPI will filter every response through it — stripping fields that aren't in the output model, validating the shape, and generating accurate documentation.

By the end of this chapter you'll have:
- A `MatchResponse` model that defines exactly what a match looks like in API responses
- A `MatchListResponse` model for the list endpoint
- `response_model` applied to every endpoint
- `response_model_exclude_none=True` to keep responses clean

## The Problem with Returning Data Directly

Here's what a GET `/matches/1` response looks like right now:

```json
{
  "id": 1,
  "home_team": "arsenal",
  "away_team": "chelsea",
  "venue": "emirates stadium",
  "date": "2025-10-21",
  "sport": "football",
  "status": "completed",
  "winner": "draw",
  "home_lineup": null,
  "away_lineup": null
}
```

Notice `"home_lineup": null` and `"away_lineup": null`. Those `null` values are noise — the lineup hasn't been set, and the client gains nothing from seeing `null` instead of the field simply not appearing. With 10 or 20 optional fields this becomes very cluttered.

More importantly, the `Match` model has `id: Optional[int] = None` — `id` starts as `None` and gets assigned when a match is created. If something goes wrong and a match with `id: null` slips into a response, the client has no idea whether `id` is supposed to be there or not, because there's no declared contract saying "this field is always an integer."

A response model solves both problems.

---

## Step 1 — Create `MatchResponse`

`MatchResponse` is a separate Pydantic model that describes the shape of a match *as seen by the API client*. It's the public contract of your API.

```python title="matches.py"
class MatchResponse(BaseModel):
    id: int
    home_team: str
    away_team: str
    venue: Optional[str] = None
    date: dt_date
    sport: Sport
    status: Status
    winner: Optional[Winner] = None
    home_lineup: Optional[List[Player]] = None
    away_lineup: Optional[List[Player]] = None
```

Compare this to `Match`:

- **`id: int`** instead of `id: Optional[int] = None` — in a response, the match always has an ID. If it doesn't, that's a bug, not an expected state.
- **No validators** — `MatchResponse` is a pure output shape. It doesn't need to validate input; FastAPI just uses it to filter and serialize.
- **No `Field(min_length=..., examples=...)` constraints** — those belong on the input model `Match`, not the output. The data coming out has already been validated going in.

<Callout type="info" title="Input Models vs Output Models">
  Think of your models in three roles:
  - **Input model** (`Match`, `MatchUpdate`) — validates what comes *in*. Has constraints, validators, optional IDs.
  - **Output model** (`MatchResponse`) — describes what goes *out*. Clean, no validation logic, required fields are truly required.
  - **Filter model** (`MatchFilters`, etc.) — shapes query parameters.

  Separating input from output is a sign of a well-designed API. It means you can change your internal data representation without accidentally changing what clients receive.
</Callout>

## Step 2 — Create `MatchListResponse`

The list endpoint returns a wrapper object with pagination metadata, not just a bare list of matches. Create a model for that shape too:

```python title="matches.py"
class MatchListResponse(BaseModel):

    page: int
    limit: int

```

`matches` is typed as `List[MatchResponse]`, not `List[Match]`. This means every match in the list will also be filtered through `MatchResponse`. You get the same clean output guarantee for the list as you do for a single match.

### What `response_model` does

When you set `response_model=MatchResponse`, FastAPI does three things automatically every time that endpoint returns data:

1. **Filters the output** — any field on the returned object that is *not* in `MatchResponse` is silently stripped. This is the leakage protection.
2. **Validates the output** — the returned data is passed through `MatchResponse`. If somehow a required field like `id` is missing, FastAPI raises an internal server error rather than sending broken data to the client.
3. **Documents the schema** — Swagger UI and the OpenAPI spec now show `MatchResponse` as the exact return type of this endpoint, giving clients an accurate contract.

### What `response_model_exclude_none=True` does

Without this flag, any field in `MatchResponse` that is `None` is still included in the JSON output as `null`. With the flag, those fields are simply omitted:

```json title="Without response_model_exclude_none=True"

  "venue": "madison square garden",

}
```

```json title="With response_model_exclude_none=True"
{
  "id": 4,
  "home_team": "knicks",
  "away_team": "celtics",
  "venue": "madison square garden",
  "date": "2026-03-05",
  "sport": "basketball",
  "status": "upcoming"
}
```

The upcoming Knicks match has no winner and no lineups yet — so there's no reason to advertise those fields as `null`. The response is cleaner and smaller.

<Callout type="info" title="null vs missing — what's the difference?">
  In JSON, `"winner": null` explicitly says "this field exists but has no value." A missing field says "this field isn't relevant right now." For optional things that genuinely don't apply yet (no lineup, no winner for upcoming matches), omitting them is more honest and makes client-side code simpler — you just check `if (match.winner)` rather than checking for `null`.
</Callout>

## Step 4 — Update the DELETE Endpoint

Previously the delete endpoint returned:

```python
return {"message": "Match deleted successfully", "match": deleted_match}
```

With `response_model=MatchResponse` set on the endpoint, that dict no longer matches the expected shape. Simplify it to just return the deleted match directly:

```python title="matches.py"
@app.delete("/matches/{match_id}", response_model=MatchResponse, response_model_exclude_none=True, tags=["Matches"])
def delete_match(match_id: Annotated[int, Path(ge=1, title="Match ID")]):
    for i, match in enumerate(INITIAL_DATA):
        if match.id == match_id:
            deleted_match = INITIAL_DATA.pop(i)
            return deleted_match  # !mark

    return {"error": "Match not found"}, 404
```

The client already knows a DELETE succeeded because we'll add a proper 200/204 status code in the next chapter. The returned match object is confirmation enough of what was deleted.

## How FastAPI Filters the Response

It's worth understanding exactly how `response_model` works under the hood — not because you need to know the internals, but because it explains a common beginner confusion: *"I'm returning a `Match` object, but the docs say `MatchResponse` — isn't that wrong?"*

No — it's correct. FastAPI doesn't care what type your function actually returns. It takes whatever value your function returns and passes it through `response_model` like a filter:

```
Your function returns:  Match object (or dict, or anything)
                              │
                              ▼
          FastAPI calls: MatchResponse.model_validate(your_return_value)
                              │
                              ▼
       Client receives:  MatchResponse-shaped JSON
```

Because `MatchResponse` and `Match` share the same field names, `model_validate` can build a `MatchResponse` from a `Match` object perfectly. If your function returned a `Match` with an extra internal field that wasn't in `MatchResponse`, that field would simply not appear in the output.

This is also why you can keep returning a plain `dict` from `list_matches`:

```python
return {
    "total": total,
    "page": pagination.page,
    "limit": pagination.limit,
    "matches": results,
}
```

FastAPI takes that dict, validates it against `MatchListResponse`, and the `matches` list (containing `Match` objects) gets each item filtered through `MatchResponse`. It all works because Pydantic's `model_validate` is flexible — it accepts dicts, other model instances, and anything that looks like the right shape.

## What You've Learned

- **`MatchResponse`** — a clean output model with `id: int` (required), no validators, no input constraints
- **`MatchListResponse`** — wraps the list with pagination metadata; uses `List[MatchResponse]` so every match in the list is also filtered
- **`response_model=`** — goes on the decorator; filters output, validates shape, documents the schema accurately
- **`response_model_exclude_none=True`** — omits `None` fields from the JSON response, keeping output clean for optional fields that don't apply yet
- **Input vs output models** — `Match` validates what comes in; `MatchResponse` describes what goes out; they're different jobs
