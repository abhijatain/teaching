In the last chapter you added `response_model` to every endpoint — your API now sends clean, predictable JSON. But there's still one big rough edge. Look at what happens when you request a match that doesn't exist:

```bash
GET /matches/999
```

```json title="Current response — wrong in two ways"
{
  "error": "Match not found"
}
```

The problem is **not** the message. The problem is the status code. This response comes back with HTTP **200 OK** — meaning "success." The client thinks the request worked fine and received valid match data, then has to inspect the body to discover the error. That's not how HTTP is supposed to work.

HTTP has an entire vocabulary of status codes specifically designed to communicate what happened. A missing resource should be **404 Not Found**. A newly created resource should be **201 Created**. A validation failure should be **422 Unprocessable Entity**. Every HTTP client, every browser, every monitoring tool, every frontend framework understands these codes — and relies on them.

By the end of this chapter you'll have:
- Replaced all `return {"error": "..."}, 404` patterns with proper `HTTPException`
- Added correct status codes to every endpoint
- Imported and used FastAPI's `status` module for readable, self-documenting code

## HTTP Status Codes — A Quick Reference

You don't need to memorise hundreds of codes. Here are the ones relevant to a CRUD API:

| Code | Name | When to use |
|------|------|-------------|
| `200` | OK | Default for GET, PATCH, PUT, DELETE — request succeeded |
| `201` | Created | Resource was successfully created (POST) |
| `204` | No Content | Success but nothing to return (sometimes used for DELETE) |
| `400` | Bad Request | The request is malformed or makes no logical sense |
| `404` | Not Found | The requested resource doesn't exist |
| `422` | Unprocessable Entity | The request is well-formed but fails business logic validation |

The distinction between `400` and `422` trips up beginners. Here's a simple way to think about it:
- `400` — "I can't even understand this request" (e.g. malformed JSON, completely wrong shape)
- `422` — "I understand the request, but the data fails validation" (e.g. upcoming match with a past date, completed match without a winner)

FastAPI already uses `422` automatically for Pydantic validation errors. You should use it too when your own business logic rejects valid-but-wrong data.

## The Problem with Returning Error Dicts

Currently the code handles "not found" like this in several places:

```python
# ❌ Current approach — wrong status code
return {"error": "Match not found"}, 404
```

This is a Python tuple being returned from a FastAPI route. FastAPI does pick up the `404` status code from the tuple — so the status code is technically right. But there are two problems:

**Problem 1 — It bypasses `response_model`.**
Remember the `response_model=MatchResponse` set on these endpoints? When you return this error dict, FastAPI tries to validate it against `MatchResponse` and either fails silently or raises an internal error, because `{"error": "Match not found"}` is not a valid `MatchResponse`. The behaviour is undefined and fragile.

**Problem 2 — The error shape is inconsistent.**
Returning raw dicts means every error in your codebase can look different. Some might use `"error"`, others `"message"`, others `"detail"`. Clients can't reliably parse your errors.

`HTTPException` solves both problems cleanly.

---

## Step 1 — Import `HTTPException` and `status`

Add two new imports:

```python title="matches.py"
# Before
from fastapi import Body, Depends, FastAPI, Path

# After
from fastapi import Body, Depends, FastAPI, HTTPException, Path, status
```

`HTTPException` is how you signal errors in FastAPI. `status` is a module containing named constants for every HTTP status code — `status.HTTP_404_NOT_FOUND`, `status.HTTP_201_CREATED`, and so on.

### Why use `status.HTTP_404_NOT_FOUND` instead of just `404`?

Both work. But the named constant is better for two reasons:

```python
# ❌ Magic number — what does 404 mean again?
raise HTTPException(status_code=404, detail="Match not found")

# ✅ Self-documenting — instantly clear
raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail="Match not found")
```

The named constant also makes typos obvious at a glance. `status.HTTP_40_NOT_FOUND` causes an `AttributeError` immediately. `staus_code=40` silently sends a weird response that's hard to debug.

---

## Step 2 — Replace Error Returns with `HTTPException`

`HTTPException` is raised, not returned. That's the key difference — `raise` immediately stops the function and sends the error response. `return` keeps executing and can cause issues if there's code after it.

Go through every endpoint and replace the tuple-return pattern:

```python title="Before — in get_match, update_match, replace_match, delete_match"
return {"error": "Match not found"}, 404
```

```python title="After"
raise HTTPException(
    status_code=status.HTTP_404_NOT_FOUND,
    detail="Match not found"
)
```

FastAPI handles the rest. When you raise `HTTPException`, FastAPI automatically:
- Sets the HTTP status code to whatever you passed
- Wraps `detail` in a consistent JSON shape: `{"detail": "Match not found"}`
- Skips `response_model` filtering (error responses aren't match data)

The `{"detail": "..."}` shape is FastAPI's standard error format. It's the same format that Pydantic validation errors use, so all your errors now look consistent.

```python title="matches.py"
        status_code=status.HTTP_404_NOT_FOUND,
        detail="Match not found"
    )
```

## Step 3 — Add `status_code` to POST (201 Created)

Right now `POST /matches` returns `200 OK` on success. That's technically not wrong, but it's imprecise. The correct code for "a new resource was created" is `201 Created`.

Add `status_code` to the POST decorator:

```python title="matches.py"
@app.post(
    "/matches",
    response_model=MatchResponse,
    response_model_exclude_none=True,
    status_code=status.HTTP_201_CREATED,  # !mark
    tags=["Matches"],
)
def create_match(...):
```

`status_code` on the decorator sets the default success status for that endpoint. When the function returns normally, FastAPI uses this code. `HTTPException` overrides it when something goes wrong.

### Also clean up `create_match`'s validation errors

The `create_match` function has some leftover manual validation checks that predate Pydantic validators. Most of them duplicate what `Match`'s `model_validator` already does — but the date check is genuinely business logic (not a schema rule), so keep it and turn it into a proper `HTTPException`:

```python title="matches.py — create_match body"
def create_match(match: ...):
    if match.status is Status.upcoming and match.date < dt_date.today():
        raise HTTPException(  # !mark
            status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,  # !mark
            detail="Upcoming match date cannot be in the past",  # !mark
        )  # !mark
    match.id = INITIAL_DATA[-1].id + 1
    INITIAL_DATA.append(match)
    return match
```

The other manual checks (`home_team == away_team`, `winner without completed status`, etc.) are already caught by `Match`'s `model_validator` before the function body even runs. FastAPI raises a `422` for those automatically, so the redundant manual checks can be removed.

<Callout type="info" title="Pydantic validation errors vs HTTPException">
  When Pydantic validation fails on request body parsing — before your function runs — FastAPI automatically returns a `422` with a detailed breakdown of every failing field. You don't control this; FastAPI handles it.

  `HTTPException` is for errors you raise yourself, after Pydantic has already accepted the data. They serve different purposes but both result in non-200 responses.
</Callout>

```json title="GET /matches?sport=baseball&status=upcoming — no results"
}
```

Notice that Pydantic's automatic `422` (the last example) has a richer `detail` — it's a list with field-level information. Your manual `HTTPException` raises have a simple string `detail`. Both are valid; FastAPI allows `detail` to be either a string or a more complex object.

## What `raise` vs `return` Means for Errors

This is worth making explicit because it's a pattern you'll use constantly:

```python
def get_match(match_id):
    for match in INITIAL_DATA:
        if match.id == match_id:
            return match              # ← success path: return the data

    raise HTTPException(              # ← error path: raise an exception
        status_code=status.HTTP_404_NOT_FOUND,
        detail="Match not found"
    )
    # Nothing after raise ever runs
```

`raise` works like `return` in one important way — it immediately exits the function. But unlike `return`, it signals to FastAPI that something went wrong and the normal response pipeline (including `response_model`) should be skipped. FastAPI catches the `HTTPException` and turns it into the error response automatically.

A useful mental model: **`return` is for success, `raise` is for failure.**

  `204 No Content` is also valid for DELETE — it says "success, and here's nothing." We're using `200` and returning the deleted match because it's useful for the client to confirm what was deleted. Both are correct; it depends on your API design preference. If you later decide not to return the deleted resource, change the status to `204` and remove `response_model`.
</Callout>

## What You've Learned

- **HTTP status codes** — `200` success, `201` created, `404` not found, `422` unprocessable entity, and when to use each
- **`HTTPException`** — FastAPI's way to signal errors; `raise` it instead of returning error dicts
- **`status` module** — named constants like `status.HTTP_404_NOT_FOUND` instead of magic numbers
- **`raise` vs `return`** — `return` for success, `raise` for failure; `raise` skips `response_model` and exits immediately
- **`status_code=` on the decorator** — sets the default success code for an endpoint (used for `201` on POST)
- **Consistent error shape** — `HTTPException` always produces `{"detail": "..."}`, making errors predictable for clients
- **400 vs 422** — `400` for malformed requests, `422` for valid-but-logically-wrong data
