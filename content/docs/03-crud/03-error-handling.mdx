## Understanding PATCH, PUT, and DELETE

Before writing any code, let's understand what each method is for:

| Method | What it does | When to use it |
|--------|-------------|----------------|
| `PATCH` | Update **some** fields | Most common — change venue, record a result, add a lineup |
| `PUT` | Replace the **entire** resource | Less common — completely overwrite a match |
| `DELETE` | Remove the resource | Remove a match permanently |

Think of it like editing a document:
- **PATCH** is like Track Changes — touch only specific parts
- **PUT** is like "Select All → Delete → Retype" — start fresh
- **DELETE** is throwing the document in the bin

## A Quick Fix First — Renaming the `date` Import

Before building the update endpoint, we need to fix a subtle but important naming conflict.

Previously the code had:

```python
from datetime import date
```

This works fine for the `Match` model where `date` is a required field. But the moment you write `MatchUpdate` — the model for partial updates — you need the field to be optional:

```python
# ❌ This breaks — Python sees 'date' and doesn't know if you mean
#    the type (datetime.date) or the field name
date: Optional[date] = None
```

Python gets confused because `date` is now used as both the **type annotation** (`datetime.date`) and the **field name** (`date`). The name on the left shadows the type on the right, causing a runtime error.

The fix is simple — import `date` under a different name using `as`:

```python
from datetime import date as dt_date
```

Now the type and the field name are clearly different things:

```python
# ✅ This is unambiguous — dt_date is the type, date is the field name
date: Optional[dt_date] = None
```

You'll also update everywhere `date` was used as a type — in `Match`, `DateRangeFilters`, and the seed data:

```python
# In Match model
date: dt_date

# In DateRangeFilters
from_date: Optional[dt_date] = None
to_date: Optional[dt_date] = None

# In seed data
date=dt_date(2025, 10, 21),

# In create_match endpoint
if match.status is Status.upcoming and match.date < dt_date.today():
```

<Callout type="warn" title="Why This Only Breaks in MatchUpdate">
  In the original `Match` model, `date: date` technically works because `date` the field name and `date` the type are the same token — Python resolves it correctly in that context. But as soon as you write `Optional[date]`, the name resolution breaks down. Aliasing to `dt_date` is the clean, unambiguous solution and is a common pattern when your field names match Python built-ins or standard library types.
</Callout>

## Another Fix — Switching from `Query()` to `Depends()`

While we're fixing things before building the update endpoints, there's one more change in the final code worth explaining: the `list_matches` function signature switched from `Query()` to `Depends()`.

**Previously the code looked like this:**

```python title="Before"
@app.get("/matches", tags=["Matches"])
def list_matches(
    filters: Annotated[MatchFilters, Query()],
    date_range: Annotated[DateRangeFilters, Query()],
    pagination: Annotated[PaginationParams, Query()],
    sort_params: Annotated[SortParams, Query()],
):
```

**Now it looks like this:**

```python title="After"
@app.get("/matches", tags=["Matches"])
def list_matches(
    filters: Annotated[MatchFilters, Depends()],
    date_range: Annotated[DateRangeFilters, Depends()],
    pagination: Annotated[PaginationParams, Depends()],
    sort_params: Annotated[SortParams, Depends()],
):
```

### Why `Query()` causes a problem

When you use `Query()` to inject a Pydantic model as query parameters, FastAPI treats the model as a flat list of required query string values. The issue is that **FastAPI with `Query()` doesn't properly respect `Optional` fields with defaults inside nested Pydantic models** — some fields can incorrectly appear as required in the generated docs, and edge cases in validation may fail.

For example, with `Query()`, a field like this:

```python
sport: Optional[Sport] = Field(None, ...)
```

...might still show up as required in Swagger UI, or throw an error if the user doesn't pass it. That's the opposite of what we want — all filter params should be genuinely optional.

### What `Depends()` does instead

`Depends()` is FastAPI's **dependency injection system**. Instead of treating the model as raw query string parsing, FastAPI calls the model's `__init__` method like a regular function, passing query parameters as keyword arguments. This means:

- All fields with default values (like `= None`) are truly optional
- Pydantic model validators run correctly
- The generated docs show the fields accurately

In short: `Depends()` is the right way to inject a Pydantic model as a group of query parameters. `Query()` is designed for individual fields, not whole models.

<Callout type="info" title="We Haven't Taught Depends() Yet — and That's Fine">
  `Depends()` is a big topic in FastAPI that deserves its own chapter. The full power of dependency injection — reusable dependencies, shared database connections, auth checks — will be covered later.

  For now, just know this: **wherever you see a Pydantic model being used as a group of query parameters, use `Depends()` instead of `Query()`**. It makes all the fields correctly optional and plays nicely with Pydantic validation. Think of it as FastAPI's way of saying "please build this object from the incoming request and hand it to my function."
</Callout>

Also note the import line changes — `Query` is removed and `Depends` is added:

```python
# Before

# After
from fastapi import Body, Depends, FastAPI, Path
```

## Step 1 — Create the `MatchUpdate` Model

For PATCH to work, you need a separate model where **every field is optional**. This is because the client only sends the fields they want to change.

```python title="matches.py"

    venue: Optional[str] = Field(
        None,
        min_length=3,
        max_length=100,
        examples=["Wembley Stadium"],
    )
    date: Optional[dt_date] = None
    status: Optional[Status] = Field(None, examples=["completed"])
    winner: Optional[Winner] = Field(None, examples=["home_team"])
    home_lineup: Optional[List[Player]] = None
    away_lineup: Optional[List[Player]] = None

    @field_validator("venue")
    @classmethod
    def normalize_venue(cls, value):
        if value is not None:
            return value.strip().lower()
        return value

    @model_validator(mode="after")
    def validate_update_logic(self):
        if self.status == Status.completed and self.winner is None:
```

Notice a few things about this model compared to `Match`:

- **Everything is `Optional`** — none of the fields are required. A client can send just one field if they want.
- **No `id`, `home_team`, `away_team`, or `sport`** — those fields define the match's identity. They shouldn't change after creation. If they need to change, delete and recreate the match.
- **Validators still apply** — if you do send `status` and `winner`, they still have to make sense together.

<Callout type="info" title="What Fields Should Be Updatable?">
  A good rule of thumb: fields that **describe the match at creation** (who's playing, what sport) shouldn't be updatable. Fields that **evolve as the match happens** (venue change, date reschedule, result, lineups) are fair game for PATCH.
</Callout>

## Step 2 — Add the PATCH Endpoint

PATCH is the most common update method. Here's the full implementation:

```python title="matches.py"
@app.patch("/matches/{match_id}", tags=["Matches"])
def update_match(
    match_id: Annotated[int, Path(ge=1, title="Match ID")],
    update: Annotated[
        MatchUpdate,
        Body(
            openapi_examples={
                "update_venue": {
                    "summary": "Update venue only",
                    "description": "Change the match venue without affecting other fields",
                    "value": {"venue": "Wembley Stadium"},
                },
                "record_result": {
                    "summary": "Record match result",
                    "description": "Update a match with completion status and winner",
                    "value": {
                        "status": "completed",
                        "winner": "home_team",
                    },
                },
                "reschedule_match": {
                    "summary": "Reschedule match date",
                    "description": "Change the date and venue for an upcoming match",
                    "value": {
                        "date": "2026-06-15",
                        "venue": "Etihad Stadium",
                    },
                },
                "add_lineup": {
                    "summary": "Add team lineup",
                    "description": "Add starting lineup before match",
                    "value": {
                        "home_lineup": [
                            {
                                "number": 1,
                                "name": "Aaron Ramsdale",
                                "position": "goalkeeper",
                                "is_captain": False,
                            },
                            {
                                "number": 10,
                                "name": "Martin Odegaard",
                                "position": "midfielder",
                                "is_captain": True,
                            },
                        ]
                    },
                },
            }
        ),
    ],
):
    """
    Partially update a match (PATCH).

    Only the fields you provide will be updated. Other fields remain unchanged.
    The update must result in a valid match state.
    """
    for i, match in enumerate(INITIAL_DATA):
        if match.id == match_id:
            # 1. Convert the existing match to a dict
            stored_match_data = match.model_dump()

            # 2. Get the update data as a dict (exclude fields the user didn't send)
            update_data = update.model_dump(exclude_unset=True)  # !mark

            # 3. Merge the dictionaries (update_data overwrites stored_match_data)
            updated_match_dict = {**stored_match_data, **update_data}  # !mark

            # 4. Re-validate to create a fresh, perfectly typed Match object
            try:
                validated_match = Match.model_validate(updated_match_dict)  # !mark
            except ValueError as e:
                return {"error": f"Update would create invalid match: {str(e)}"}, 400

            # 5. Save the validated model back to our "database"
            INITIAL_DATA[i] = validated_match

            return validated_match

    return {"error": "Match not found"}, 404
```

Let's walk through each step in the logic:

**Step 1 — Find the match using `enumerate`**

```python
for i, match in enumerate(INITIAL_DATA):
    if match.id == match_id:
```

You might wonder: why use `enumerate` instead of a plain `for match in INITIAL_DATA`? 

The answer is that we need to **replace** the match in the list later (Step 5), and to replace an item in a list you need to know its **position** (index). `enumerate` gives you both the index `i` and the item `match` at the same time. Without `i`, you'd find the match but have no way to write back to the correct slot.

**Step 2 — Dump the existing match to a dict**

```python
stored_match_data = match.model_dump()
# Result: {"id": 1, "home_team": "arsenal", "date": "2025-10-21", "status": "upcoming", ...}
```

We convert the stored Pydantic `Match` object into a plain Python dictionary. We need a dict because we can't directly merge two Pydantic model objects — but we can merge two dicts. This gives us a snapshot of the current state of the match with all its fields.

**Step 3 — Get only the fields the user actually sent**

```python
update_data = update.model_dump(exclude_unset=True)
# If user sent {"venue": "Wembley Stadium"}, this is {"venue": "wembley stadium"}
# NOT {"venue": "wembley stadium", "date": None, "status": None, ...}
```

`exclude_unset=True` is the key to partial updates. Without it, `model_dump()` returns **every** field in `MatchUpdate` — all the ones the user didn't send come back as `None`. If you then merge that into the stored data, you'd overwrite real values with `None` and destroy data the user never intended to touch.

<Callout type="warn" title="The exclude_unset=True Magic">
  ```python
  # ❌ Wrong — this wipes everything the user didn't send
  update.model_dump()
  # {"venue": "wembley stadium", "date": None, "status": None, "winner": None, ...}

  # ✅ Right — only the fields the user actually provided
  update.model_dump(exclude_unset=True)
  # {"venue": "wembley stadium"}
  ```
</Callout>

**Step 4 — Merge old data with new data**

```python
updated_match_dict = {**stored_match_data, **update_data}
```

The `**` syntax is called **dictionary unpacking**. It spreads all the key-value pairs from a dictionary into a new one. When two dicts are unpacked together, the second one wins on any keys they share.

Here's a concrete example:

```python
stored = {"id": 1, "venue": "emirates stadium", "status": "upcoming", "date": "2025-10-21"}
update = {"venue": "wembley stadium"}

merged = {**stored, **update}
# {"id": 1, "venue": "wembley stadium", "status": "upcoming", "date": "2025-10-21"}
#                   ^^^ overwritten                ^^^ kept      ^^^ kept
```

Only `venue` changed. Everything else stays exactly as it was.

**Step 5 — Re-validate the complete merged match**

```python
try:
    validated_match = Match.model_validate(updated_match_dict)
except ValueError as e:
    return {"error": f"Update would create invalid match: {str(e)}"}, 400
```

This is the most important step. `MatchUpdate` validators only check that the *update request itself* is reasonable. But they can't see the full picture of the match. Consider this scenario:

```json title="PATCH /matches/4 — match currently has no winner"
{ "status": "completed" }
```

`MatchUpdate` allows this request — there's nothing wrong with sending just `status`. But when you merge it with the stored match (which has `winner: null`), the final result is a completed match without a winner — which violates the `Match` model's rules.

`Match.model_validate(updated_match_dict)` catches exactly this. It runs **all** the `Match` validators on the fully merged dictionary, just like they'd run when creating a brand new match. If the combined result would be invalid, it throws a `ValueError` which we catch and return as a 400 error to the client.

**Step 6 — Save the validated match back to the list**

```python
INITIAL_DATA[i] = validated_match
return validated_match
```

We use the index `i` (saved earlier from `enumerate`) to replace the match at its exact position in `INITIAL_DATA`. This is why we needed `enumerate` in Step 1 — `INITIAL_DATA[i] = validated_match` writes back to the right slot. Then we return the freshly validated match object so the client sees exactly what was saved.

## Step 3 — Add the PUT Endpoint (Full Replacement)

PUT replaces the entire match. The client must send every required field — there's no merging with existing data.

```python title="matches.py"
@app.put("/matches/{match_id}", tags=["Matches"])
def replace_match(
    match_id: Annotated[int, Path(ge=1, title="Match ID")],
    match: Annotated[
        Match,
        Body(
            openapi_examples={
                "replace_upcoming": {
                    "summary": "Replace with upcoming match",
                    "description": "Completely replace match data with new upcoming match details",
                    "value": {
                        "home_team": "Manchester City",
                        "away_team": "Tottenham",
                        "sport": "football",
                        "date": "2026-05-20",
                        "status": "upcoming",
                        "venue": "Etihad Stadium",
                    },
                },
                "replace_completed": {
                    "summary": "Replace with completed match",
                    "description": "Replace entire match with completed match data including winner",
                    "value": {
                        "home_team": "Arsenal",
                        "away_team": "Chelsea",
                        "sport": "football",
                        "date": "2026-05-15",
                        "status": "completed",
                        "venue": "Wembley Stadium",
                        "winner": "home_team",
                    },
                },
            }
        ),
    ],
):
    """
    Replace an entire match with new data (PUT).

    All fields must be provided. This completely replaces the existing match.
    Use PATCH if you only want to update specific fields.
    """
    for i, existing_match in enumerate(INITIAL_DATA):
        if existing_match.id == match_id:
            match.id = match_id  # Preserve the ID  # !mark
            INITIAL_DATA[i] = match
            return match

    return {"error": "Match not found"}, 404
```

Notice the differences from PATCH:

- The body parameter is `match: Match`, not `match: MatchUpdate`. The client sends a complete `Match` — all required fields must be present.
- There's no merging. The incoming `match` fully replaces what's stored.
- We **preserve the ID** from the URL path (`match.id = match_id`). The client shouldn't be able to change an ID just by sending a different value in the body.

Try it with an incomplete body and FastAPI will reject it immediately — missing required fields are a validation error before your function even runs.

```json title="PUT /matches/1 — all required fields must be present"
{
  "home_team": "Arsenal",
  "away_team": "Chelsea",
  "sport": "football",
  "date": "2026-05-15",
  "status": "completed",
  "venue": "Wembley Stadium",
  "winner": "home_team"
}
```

<Callout type="info" title="PUT vs PATCH in Practice">
  Most modern APIs only implement PATCH. Full replacement is rarely what users need — it's easy to accidentally wipe a field you forgot to include. That said, PUT is useful when you want an explicit "replace everything" contract, like a settings page where the user always submits the full form. This tutorial shows you both so you can choose what fits your API.
</Callout>

## Step 4 — Add the DELETE Endpoint

DELETE removes a match by ID and returns the deleted data so the client knows what was removed.

```python title="matches.py"
@app.delete("/matches/{match_id}", tags=["Matches"])
def delete_match(match_id: Annotated[int, Path(ge=1, title="Match ID")]):
    """
    Delete a match by ID.

    Returns the deleted match data on success.
    """
    for i, match in enumerate(INITIAL_DATA):
        if match.id == match_id:
            deleted_match = INITIAL_DATA.pop(i)
            return {"message": "Match deleted successfully", "match": deleted_match}

    return {"error": "Match not found"}, 404
```

The logic is straightforward:

1. Loop through `INITIAL_DATA` using `enumerate` to get both the index `i` and the match object.
2. When the ID matches, call `INITIAL_DATA.pop(i)` — this removes the item at position `i` from the list and returns it.
3. Return a confirmation message along with the deleted match data.
4. If no match is found, return a 404.

Why return the deleted match? It gives the client a chance to confirm what was deleted without needing a separate GET call first. This is a common REST convention.

Test it:

```bash

```


## Two-Layer Validation: Why It Matters

The PATCH implementation uses two separate layers of validation and it's worth understanding why both are needed. Looking back at Steps 3 and 5 of the PATCH logic:

```
Request body arrives
       │
       ▼
┌─────────────────────┐
│  MatchUpdate model  │  ← Layer 1: Are the update fields valid on their own?
│  (field validators  │    e.g. "Is status a valid Status enum value?"
│   + model validator)│    e.g. "If setting status=completed, is winner also set?"
└─────────────────────┘
       │
       ▼
  Merge with existing match data
       │
       ▼
┌─────────────────────┐
│  Match.model_validate│  ← Layer 2: Is the resulting complete match valid?
│  (full Match rules) │    e.g. "A completed match without a winner is invalid"
└─────────────────────┘
       │
       ▼
  Save + return
```

**Layer 1** (`MatchUpdate` validators) catches problems *within the update itself* — like trying to set `status=completed` without providing a `winner` in the same update request.

**Layer 2** (`Match.model_validate`) catches problems in the *final state* — like setting `status=completed` when the stored match already had a `winner=None` that wasn't part of the update.

Without Layer 2, you could sneak a match into an invalid state through a carefully crafted partial update.

## PATCH Implementation: The Merge Pattern

The merge-then-revalidate pattern is worth keeping in your toolkit. Here it is stripped down:

```python title="The PATCH merge pattern"
# 1. Load the stored resource
stored_data = existing_resource.model_dump()

# 2. Get only what the client sent
update_data = update.model_dump(exclude_unset=True)

# 3. Merge: update_data wins on conflicts
merged = {**stored_data, **update_data}

# 4. Revalidate the full result
try:
    validated = FullModel.model_validate(merged)
except ValueError as e:
    return {"error": str(e)}, 400

# 5. Save
resource_store[index] = validated
return validated
```

This pattern works for any resource, not just matches.

## What You've Learned

- **`from datetime import date as dt_date`** — aliasing the import prevents a name collision when using `date` as both a field name and a type annotation in `Optional[dt_date]`
- **`Depends()` vs `Query()`** — use `Depends()` when injecting a whole Pydantic model as a group of query parameters; it correctly makes all optional fields optional
- **`MatchUpdate`** — a separate model where all fields are `Optional`, used for PATCH
- **`exclude_unset=True`** — returns only the fields the client actually sent, preventing accidental overwrites
- **`enumerate`** — gives you both the index and the item so you can write back to the correct position in the list
- **`{**a, **b}` dict merge** — unpacks both dicts; the second dict wins on overlapping keys
- **Merge + revalidate** — the correct pattern for applying partial updates safely
- **Two-layer validation** — `MatchUpdate` validators check the update itself; `Match.model_validate` checks the final merged state
- **PATCH** — partial update, most common
- **PUT** — full replacement, less common but useful
- **DELETE** — removes a resource and returns confirmation
