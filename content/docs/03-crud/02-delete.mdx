# Deleting Resources (DELETE)

You've mastered creating (POST), reading (GET), and updating (PUT/PATCH) resources. The final piece of CRUD is **DELETE** — removing resources from your system. In this lesson, you'll learn how to implement safe deletion endpoints with proper validation, error handling, and important production considerations.

---

## Where We Are

Your API now supports the CRU in CRUD:
- **Create:** `POST /matches` — Add new matches
- **Read:** `GET /matches` and `GET /matches/{id}` — Retrieve matches
- **Update:** `PUT /matches/{id}` and `PATCH /matches/{id}` — Modify matches

The final operation is **Delete** — removing matches that shouldn't exist anymore.

---

## Step 1 — Basic DELETE Endpoint

Start with the simplest form: delete a single resource by ID.

```python title="matches.py"

    match_id: Annotated[
        int,
        Path(
            ge=1,
            title="Match ID",
            description="ID of the match to delete",
        ),
    ],
):
    """
    Delete a match by ID.
    
    Returns the deleted match data for confirmation.
    """
    for i, match in enumerate(INITIAL_DATA):
        if match.id == match_id:
            deleted_match = INITIAL_DATA.pop(i)
            return {
                "message": "Match deleted successfully",
                "deleted_id": match_id,
                "deleted_match": deleted_match,
            }

```

Breaking this down:
- `@app.delete()` — Defines a DELETE endpoint
- `INITIAL_DATA.pop(i)` — Removes the match from the list and returns it
- We return the deleted match so the user can verify what was deleted
- 404 if the match doesn't exist

<Callout type="info" title="Why Return the Deleted Resource?">
Returning the deleted resource serves two purposes:
1. **Confirmation** — User can verify they deleted the right thing
2. **Undo capability** — Client can keep the data to potentially restore it

Some APIs return just `{"message": "deleted"}` or even an empty response with `204 No Content`. Both approaches are valid.
</Callout>

## Step 4 — Add Safety Checks (Optional Validation)

Sometimes you want to prevent deletion based on business rules. For example, maybe you can't delete completed matches:

```python title="matches.py - adding business logic"
@app.delete("/matches/{match_id}", tags=["Matches"])
def delete_match(
    match_id: Annotated[int, Path(ge=1)],
):
    """Delete a match by ID."""
    for i, match in enumerate(INITIAL_DATA):
        if match.id == match_id:
            # Business rule: Can't delete completed matches  # !mark
            if match.status == Status.completed:  # !mark
                return {  # !mark
                    "error": "Cannot delete completed matches",  # !mark
                    "reason": "Historical data must be preserved",  # !mark
                }, 409  # Conflict  # !mark
            
            deleted_match = INITIAL_DATA.pop(i)
            return DeleteResponse(
                message="Match deleted successfully",
                deleted_id=match_id,
                deleted_match=deleted_match,
            )
    
    return {"error": "Match not found"}, 404
```

The `409 Conflict` status code indicates the request is valid but can't be completed due to the resource's current state.

<Callout type="warn" title="Business Rules in DELETE">
Consider carefully what can and can't be deleted:
- Historical records (completed matches, finalized orders)?
- Resources referenced by other data?
- Resources tied to payments or legal obligations?

Some systems use "soft deletes" (marking as deleted without removing) for these cases.
</Callout>

## DELETE Best Practices

### 1. **Idempotency**

DELETE should be idempotent — calling it multiple times has the same effect as calling it once:

```python
DELETE /matches/1  →  200 OK (deleted)
DELETE /matches/1  →  404 Not Found (already deleted)
DELETE /matches/1  →  404 Not Found (still doesn't exist)
```

Your implementation handles this correctly — deleting a non-existent resource returns 404, not an error.

### 2. **Soft Delete vs Hard Delete**

**Hard Delete** (what we've implemented):
```python
INITIAL_DATA.pop(i)  # Actually removes from data
```

**Soft Delete** (mark as deleted):
```python
match.deleted = True  # Still in data, just marked
match.deleted_at = datetime.now()
```

Use soft delete when:
- You need audit trails
- Data might need restoration
- Other resources reference this one
- Legal/compliance requires retention

Use hard delete when:
- Data is truly temporary (sessions, caches)
- Privacy regulations require actual deletion (GDPR "right to be forgotten")
- You're confident it should never come back

### 3. **Cascading Deletes**

What happens when you delete a match that has:
- Events (goals, cards)?
- Statistics?
- References from other tables?

Options:
```python
# Option 1: Prevent delete if dependencies exist
if match.events or match.statistics:
    return {"error": "Cannot delete match with events/stats"}, 409

# Option 2: Delete dependencies too (cascade)
match.events.clear()
match.statistics = None
INITIAL_DATA.pop(i)

# Option 3: Set dependencies to null
for event in all_events:
    if event.match_id == match_id:
        event.match_id = None
```

This tutorial doesn't enforce cascading rules because we're using an in-memory list. In a real database, you'd configure this at the schema level.

### 4. **Confirmation for Critical Deletes**

For important resources, require explicit confirmation:

```python
@app.delete("/matches/{match_id}")
```
