# Nested Models

So far our `Match` model has been flat — just a single level of fields. But real sports data is hierarchical: matches have lineups that contain individual players with their own attributes. In this lesson, we'll build complex data structures by nesting Pydantic models inside each other.

---

## Where We Are

Your current `Match` model from lesson 5 looks like this:

```python title="matches.py"
class Match(BaseModel):
    id: int
    home_team: str = Field(min_length=2, max_length=50)
    away_team: str = Field(min_length=2, max_length=50)
    venue: Optional[str] = Field(None, min_length=3, max_length=100)
    date: date
    sport: Sport
    status: Status
    winner: Optional[Winner] = None

    @field_validator('home_team', 'away_team')
    @classmethod
    def normalize_team_name(cls, value):
        return value.strip().lower()

    @field_validator('venue')
    @classmethod
    def normalize_venue(cls, value):
        if value is not None:
            return value.strip().lower()
        return value
```

This works for basic match data. But what if you want to track which players started in each match? You could add fields like `home_player_1`, `home_player_2`, etc., but that's brittle and doesn't scale. Instead, we'll use **nested models** — a `Player` model inside the `Match` model.

<Callout type="info" title="Why Nested Models?">
  Nested models let you represent complex relationships while keeping validation at each level. A `Player` isn't just a string — it's an entity with a jersey number, name, position, and captain status. Each of those fields needs its own validation rules.
</Callout>

This model uses everything you've learned:

- `Field(min_length=2, max_length=50)` — player names with sensible bounds

<Callout type="info" title="Model Order Matters">
  Define `Player` before `Match` because `Match` will reference it. Python needs to know what `Player` is before it can use it as a type annotation.
</Callout>

- `0` — the first player in the list (zero-indexed)
- `number` — the field that failed validation

This precision makes debugging easy. The error points exactly to the problematic field, even deep inside nested structures.

## How Nested Validation Works

When you send a request with nested models, validation happens layer by layer:

```python title="Validation flow"
Match(
    home_team="Arsenal",
    home_lineup=[
        Player(
            number=100,  # ❌ Invalid - exceeds le=99
            name="Aaron Ramsdale",
            position="goalkeeper"
        )
    ]
)
```

1. Pydantic validates `Match` fields (`home_team`, `sport`, `date`)
2. Pydantic sees `home_lineup` is a `List[Player]` — starts validating the list
3. For each item in the list, Pydantic validates it as a `Player`
4. `Player` field validation runs — **fails** because `number=100` exceeds `le=99`
5. Error propagates up with full path: `["body", "home_lineup", 0, "number"]`

If validation succeeds at all levels, you get a fully validated nested structure. If it fails at any level, you get a clear error message with the exact path to the problem.

<Callout type="info" title="Validation Is Recursive">
  Nested models aren't just organizational — each level has its own field validators. Pydantic validates the entire tree, not just the top level. This is what makes nested models powerful: you can enforce constraints at every level of your data structure.
</Callout>

## Optional vs Required Nested Fields

You made `home_lineup` optional because not all matches have lineup data recorded. But what if lineups are required for certain match types? You can make them non-optional:

```python title="Hypothetical - required lineups"
class Match(BaseModel):
    # ... other fields ...
    home_lineup: List[Player]  # Not Optional - must be provided
    away_lineup: List[Player]  # Not Optional - must be provided
```

Now every match creation request must include both lineups. If either is missing, FastAPI returns a validation error.

You can also require non-empty lists:

```python title="Hypothetical - at least one player required"
class Match(BaseModel):
    # ... other fields ...
    home_lineup: List[Player] = Field(min_length=1)
    away_lineup: List[Player] = Field(min_length=1)
```

This ensures that if lineups are provided, they're not empty arrays.

<Callout type="warn" title="Choose Defaults Carefully">
  Think about what makes sense for your API. If most matches don't have lineups, make them optional. If lineups are essential, make them required. The right choice depends on how your API will be used, not arbitrary preferences.
</Callout>

---

## When to Use Nested Models

✅ **Use nested models when:**
- Data has a clear hierarchical structure (match → lineup → players)
- Related fields always appear together (players always have number, name, position)
- You need validation at multiple levels (player numbers, positions)
- You want type safety and editor autocomplete

❌ **Keep models flat when:**
- Fields are truly independent (team name doesn't relate to venue)
- You'd only nest one level deep with 1-2 fields (just use separate fields)
- Data structure varies wildly between requests (consider separate models instead)

<Callout type="info" title="Nesting Depth">
  You can nest as deep as needed, but 2-3 levels is typical. Beyond 4 levels, consider whether you're modeling the right relationships — deeply nested structures can signal that separate API endpoints might be clearer.
</Callout>
