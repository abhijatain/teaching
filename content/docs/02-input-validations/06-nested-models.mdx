# Nested Models

So far our `Match` model has been flat — just a single level of fields. But real sports data is hierarchical: matches have lineups (which contain players), events (goals, cards, substitutions), and statistics (shots, possession, fouls). In this lesson, we'll build complex data structures by nesting Pydantic models inside each other.

---

## Where We Are

Your current `Match` model looks like this:

```python title="matches.py"
class Match(BaseModel):
    id: int
    home_team: str = Field(min_length=2, max_length=50)
    away_team: str = Field(min_length=2, max_length=50)
    venue: Optional[str] = Field(None, min_length=3, max_length=100)
    date: date
    sport: Sport
    status: Status
    winner: Optional[Winner] = None
    score_home: Optional[int] = Field(None, ge=0)
    score_away: Optional[int] = Field(None, ge=0)
```

This works for basic match data. But what if you want to track:
- **Players** in each lineup with jersey numbers and positions
- **Events** that happened during the match (goals, cards, substitutions)
- **Statistics** like possession percentage, shots, and fouls

You could try adding more flat fields:

```python title="Flat approach — doesn't scale"
class Match(BaseModel):
    # ... existing fields ...
    player_1_name: str
    player_1_number: int
    player_1_position: str
    player_2_name: str
    player_2_number: int
    player_2_position: str
    # ... repeat for 22+ players? 😱
```

This becomes unmanageable fast.

<Callout type="warn" title="Why Flat Models Don't Scale">
  With flat fields you can't:
  - Validate related data together (player number with player name)
  - Have variable numbers of items (some teams have 11 players, some have substitutes)
  - Reuse structures (both teams need the same player fields)
  - Keep the code readable
</Callout>

## Step 1 — Create a Player Model

Instead of adding player fields to `Match`, create a separate `Player` model:

```python title="matches.py - add this before Match"
class Player(BaseModel):
    number: int = Field(ge=1, le=99, description="Jersey number")
    name: str = Field(min_length=2, max_length=50)
    position: Literal["goalkeeper", "defender", "midfielder", "forward"]
    is_captain: bool = False
```

This is a standalone Pydantic model with its own validation rules. Notice:
- `number` is constrained to 1-99 (standard jersey numbers)
- `position` uses `Literal` to allow only four valid values
- `is_captain` has a default of `False`

Now we can use this model as a field type in `Match`:

```diff title="matches.py"
  class Match(BaseModel):
      id: int
      home_team: str = Field(min_length=2, max_length=50)
      away_team: str = Field(min_length=2, max_length=50)
      venue: Optional[str] = Field(None, min_length=3, max_length=100)
      date: date
      sport: Sport
      status: Status
      winner: Optional[Winner] = None
      score_home: Optional[int] = Field(None, ge=0)
      score_away: Optional[int] = Field(None, ge=0)
+     home_lineup: Optional[List[Player]] = None
+     away_lineup: Optional[List[Player]] = None
```

`Optional[List[Player]]` means:
- The field can be `None` (lineup not provided yet)
- If provided, it must be a list
- Every item in that list must be a valid `Player`

<Callout type="info" title="How Pydantic Validates Nested Models">
  When you send a request with nested data, Pydantic:
  1. Validates the outer `Match` fields first
  2. Then validates each `Player` in `home_lineup`
  3. Then validates each `Player` in `away_lineup`
  
  If any player has an invalid jersey number or position, the entire request is rejected with a clear error pointing to exactly which player failed validation.
</Callout>

## Step 2 — Add Match Events

Real matches have timestamped events: goals, cards, substitutions. Create another nested model:

```python title="matches.py - add this before Match"
class MatchEvent(BaseModel):
    minute: int = Field(ge=0, le=120, description="Minute when event occurred")
    event_type: Literal["goal", "yellow_card", "red_card", "substitution"]
    team: Literal["home", "away"]
    player_name: str = Field(min_length=2, max_length=50)
    description: Optional[str] = Field(None, max_length=200)
```

Then add it to `Match`:

```diff title="matches.py"
  class Match(BaseModel):
      # ... existing fields ...
      home_lineup: Optional[List[Player]] = None
      away_lineup: Optional[List[Player]] = None
+     events: List[MatchEvent] = Field(default_factory=list)
```

Notice the different syntax: `Field(default_factory=list)` instead of `None`.

Breaking this down:
- `events: List[MatchEvent]` — this field is a list of events, always present (not `Optional`)
- `= Field(default_factory=list)` — if not provided, default to an empty list `[]`

Why `default_factory=list` instead of `= []`?

```python title="Why default_factory is needed"
# ❌ Wrong — all Match instances would share the same list!
events: List[MatchEvent] = []

# ✅ Correct — each Match gets its own new list
events: List[MatchEvent] = Field(default_factory=list)
```

In Python, mutable default arguments (like `[]` or `{}`) are shared across all instances. `default_factory` creates a new list for each `Match` instance.

## Step 3 — Add Match Statistics (Nested Model Inside Nested Model)

Statistics have structure too: possession, shots, fouls — and you need these for *both* teams. This is a perfect case for a nested model inside a nested model.

First, create a model for one team's stats:

```python title="matches.py - add before Match"
class TeamStats(BaseModel):
    possession_percent: int = Field(ge=0, le=100)
    shots: int = Field(ge=0)
    shots_on_target: int = Field(ge=0)
    corners: int = Field(ge=0)
    fouls: int = Field(ge=0)

    @model_validator(mode='after')
    def validate_shots(self):
        if self.shots_on_target > self.shots:
            raise ValueError("shots_on_target cannot exceed total shots")
        return self
```

This model includes cross-field validation: shots on target can't exceed total shots.

Then create a model that wraps both teams' stats:

```python title="matches.py - add before Match"
class MatchStatistics(BaseModel):
    home_stats: TeamStats
    away_stats: TeamStats
    attendance: Optional[int] = Field(None, ge=0, description="Number of spectators")
    referee: Optional[str] = Field(None, min_length=2, max_length=50)
```

Finally, add it to `Match`:

```diff title="matches.py"
  class Match(BaseModel):
      # ... existing fields ...
      home_lineup: Optional[List[Player]] = None
      away_lineup: Optional[List[Player]] = None
      events: List[MatchEvent] = Field(default_factory=list)
+     statistics: Optional[MatchStatistics] = None
```

Now you have three levels of nesting:
1. `Match` contains `statistics`
2. `statistics` (type `MatchStatistics`) contains `home_stats` and `away_stats`
3. Each of those is a `TeamStats` with validated fields

## How Nested Validation Works

When you send a request with nested models, validation happens in order:

```python title="Validation flow"
Match(
    home_team="Arsenal",
    statistics=MatchStatistics(
        home_stats=TeamStats(
            shots=10,
            shots_on_target=15  # ❌ Invalid
        )
    )
)
```

1. Pydantic validates `Match` fields (`home_team`, etc.)
2. Pydantic sees `statistics` is a `MatchStatistics` — validates its fields
3. Pydantic sees `home_stats` is a `TeamStats` — validates its fields
4. `TeamStats` field validation passes (`shots=10` is valid)
5. `TeamStats` model validator runs — **fails** because `shots_on_target > shots`
6. Error propagates up with full path: `["body", "statistics", "home_stats"]`

If validation succeeds at all levels, you get a fully validated nested structure. If it fails at any level, you get a clear error message with the exact path to the problem.

<Callout type="info" title="Validation Is Recursive">
  Nested models aren't just organizational — each level has its own field validators and model validators. Pydantic validates the entire tree, not just the top level. This is what makes nested models powerful: you can enforce constraints at every level of your data structure.
</Callout>

## The Complete Match Model

Here's what your `Match` model looks like now with all the nested structures:

```python title="matches.py"
class Match(BaseModel):
    id: int
    home_team: str = Field(min_length=2, max_length=50)
    away_team: str = Field(min_length=2, max_length=50)
    venue: Optional[str] = Field(None, min_length=3, max_length=100)
    date: date
    sport: Sport
    status: Status
    winner: Optional[Winner] = None
    score_home: Optional[int] = Field(None, ge=0)
    score_away: Optional[int] = Field(None, ge=0)
    
    # Nested structures
    home_lineup: Optional[List[Player]] = None
    away_lineup: Optional[List[Player]] = None
    events: List[MatchEvent] = Field(default_factory=list)
    statistics: Optional[MatchStatistics] = None
```

Each nested field brings its own validation:
- `home_lineup` and `away_lineup` — each player's number, name, and position are validated
- `events` — each event's minute, type, and team are validated
- `statistics` — possession percentages, shot counts, and the shots-on-target rule are validated

And it all happens automatically before your endpoint function runs.

## When to Use Nested Models

✅ **Use nested models when:**
- Data has a clear hierarchical structure (match → lineup → players)
- Related fields always appear together (team stats are meaningless without both teams)
- You need validation at multiple levels (player numbers, event minutes, shot counts)
- You want type safety and editor autocomplete

❌ **Keep models flat when:**
- Fields are truly independent (team name doesn't relate to venue)
- You'd only nest one level deep (just use optional fields)
- Data structure varies wildly between requests (consider separate models instead)

<Callout type="info" title="Nesting Depth">
  You can nest as deep as needed, but 2-3 levels is typical. Beyond 4 levels, consider whether you're modeling the right relationships — deeply nested structures can signal that separate API endpoints might be clearer.
</Callout>
