# Query Parameter Models

At the end of the last two lessons, our `list_matches` function has eleven parameters and three manual validation checks scattered through the function body:

```python title="matches.py - where we left off"

    sport: Annotated[Optional[Sport], Query(...)] = None,

    team_name: Annotated[Optional[str], Query(min_length=3, max_length=50)] = None,
    team_filter: Annotated[Optional[TeamFilter], Query()] = None,
    from_date: Annotated[Optional[date], Query(alias="from-date")] = None,
    to_date: Annotated[Optional[date], Query(alias="to-date")] = None,
    page: Annotated[int, Query(ge=1)] = 1,
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
    sort_by: Annotated[Optional[Literal[...]], Query()] = None,
    sort_order: Annotated[Literal["asc", "desc"], Query()] = "asc",
):
    # Cross-parameter checks — all living inside the function
    if team_filter and winner:
        return {"Error": "team_filter and winner cannot be selected together."}
    if (team_filter or winner) and status is not Status.completed and status is not None:

    if from_date and to_date and from_date > to_date:
        return {"error": "from-date must be before to-date"}
```

There are two real problems as the API grows.

**The signature is hard to read.** Eleven parameters in one function is a lot to scan. It's not obvious that `page` and `limit` belong together, or that `from_date` and `to_date` are a pair.

**The cross-parameter validation is in the wrong place.** The three `if` checks at the top of the function are logically about "what makes these parameter combinations valid" — but they're buried inside filtering code. If you ever add another endpoint that accepts the same parameters, you'd have to copy all three checks there too, and keep them in sync forever.

This lesson solves both problems.

---

## The Idea: Group Parameters into Models

FastAPI lets you pass an entire Pydantic model as a query parameter group. Instead of declaring each parameter individually in the function signature, you define a model class and FastAPI automatically reads each of its fields from the query string.

The `list_matches` signature goes from eleven individual parameters to four clean groups:

```python title="matches.py - what we're building toward"
def list_matches(
    filters: Annotated[MatchFilters, Query()],
    date_range: Annotated[DateRangeFilters, Query()],
    pagination: Annotated[PaginationParams, Query()],
    sorting: Annotated[SortParams, Query()],
):
```

The URL still works exactly the same — users still send `?sport=football&page=2&from_date=2025-01-01`. FastAPI reads those values and populates the right model fields automatically. Nothing changes from the caller's perspective.

---

## Step 1 — Add Field, model_validator, field_validator, and computed_field to Imports

The models use several tools from Pydantic that we haven't imported yet. Here's what each one does before we start using them:

- **`Field()`** — the Pydantic equivalent of `Query()`, used inside model classes instead of function signatures. It's how you attach defaults, constraints, and documentation to a model field.
- **`model_validator`** — lets you write a method that checks *relationships between fields*, like "these two fields can't both be set at the same time". It runs after Pydantic has parsed all the individual fields.
- **`field_validator`** — lets you write a method that transforms or validates a *single field's value*, like stripping whitespace or converting to lowercase. It runs once for that specific field.
- **`computed_field`** — lets you add a read-only property whose value is calculated from other fields. Pydantic treats it like any other field, but you never set it manually — it's always derived.

```diff title="matches.py"
- from pydantic import BaseModel
+ from pydantic import BaseModel, Field, computed_field, field_validator, model_validator
```

## Step 2 — Add Validators to the Match Model

Before we build the filter models, there's something to fix in `Match` itself. Right now, when a match is created, team names and venue are stored as-is — `"Arsenal"`, `"ARSENAL"`, and `"arsenal"` would all be stored differently. That makes matching later fragile: you'd have to call `.lower()` everywhere you compare.

The cleaner approach is to normalize at the point of storage. Add `field_validator` methods to `Match` so that `home_team`, `away_team`, and `venue` are always lowercased when stored. Then the filtering code never needs to call `.lower()` at all.

```diff title="matches.py"
  class Match(BaseModel):
      id: int
      home_team: str
      away_team: str
      venue: Optional[str] = None
      date: date
      sport: Sport
      status: Status
      winner: Optional[Winner] = None
+
+     @field_validator('home_team', 'away_team')
+     @classmethod
+     def normalize_team_name(cls, value):
+         return value.strip().lower()
+
+     @field_validator('venue')
+     @classmethod
+     def normalize_venue(cls, value):
+         if value is not None:
+             return value.strip().lower()
+         return value
```

A few things to note. `@field_validator('home_team', 'away_team')` — you can pass multiple field names to the same validator, and Pydantic will run it for each one. `@classmethod` is required by Pydantic for field validators — it receives `cls` (the class itself) as the first argument, not `self` (an instance).

`venue` gets its own validator because it's optional — it can be `None`, so we have to check before lowercasing. `home_team` and `away_team` are always present, so no `None` check is needed.

Now the `INITIAL_DATA` entries can be written in normal casing — the validator normalizes them when the `Match` objects are created:

```python
Match(id=1, home_team="Arsenal", away_team="Chelsea", ...)
# stored as: home_team="arsenal", away_team="chelsea"
```

### Moving the conflict checks into the model

Now for the key move. These two checks currently sit at the top of `list_matches`:

```python title="matches.py - currently in the function body"
if team_filter and winner:
    return {"Error": "team_filter and winner cannot be selected together."}
if (team_filter or winner) and status is not Status.completed and status is not None:
    return {"Error": "team_filter or winner can only be used when status is 'completed'."}
```

They're about what makes a valid combination of filter fields. They belong in `MatchFilters`. Add a `model_validator` and a `field_validator` for the team name:

```diff title="matches.py"
  class MatchFilters(BaseModel):
      sport: Optional[Sport] = Field(None, title="Sport Type", ...)
      status: Optional[Status] = Field(None, title="Match Status", ...)
      winner: Optional[Winner] = Field(None, title="Winner", ...)
      team: Optional[str] = Field(None, min_length=3, max_length=50, pattern="...", title="Team Name", ...)
      team_filter: Optional[TeamFilter] = Field(None, title="Team Result Filter", ...)
+
+     @field_validator('team')
+     @classmethod
+     def normalize_team(cls, value):
+         if value is not None:
+             return value.strip().lower()
+         return value
+
+     @model_validator(mode='after')
+     def validate_winner_and_team_filter(self):
+         if self.team_filter and self.winner:
+             raise ValueError("team_filter and winner cannot be used together")
+         if (self.team_filter or self.winner) and self.status not in (Status.completed, None):
+             raise ValueError("team_filter and winner can only be used when status is 'completed'")
+         return self
```

The `field_validator` runs first — Pydantic processes individual field validators before the model validator. So by the time `validate_winner_and_team_filter` runs, `self.team` is already stripped and lowercased.

`@model_validator(mode='after')` tells Pydantic to run this method after all fields have been parsed. At that point `self` holds the fully populated model, so you can check any combination of fields directly.

When a check fails, `raise ValueError` — Pydantic catches it and FastAPI returns the same structured error response the user has seen from `Query` and `Path` validations in the previous lessons. When everything is valid, you return `self`.

<Callout type="info" title="mode='after'">
  The validator runs after Pydantic has already parsed and type-checked every individual field. This matters here because you need `self.status` to already be a `Status` enum — not a raw string — before you can compare it to `Status.completed`. `mode='before'` would run before parsing, when fields are still raw values coming in from the URL.
</Callout>

## Step 5 — Create the PaginationParams Model

`page` and `limit` go together — both control where in the result set the user is. This is also the model where the `computed_field` we covered at the end of this lesson actually lives in the real code — we'll add `offset` here so the function body can use it directly:

First add `computed_field` to the Pydantic import (if you added all four in Step 1, you're already done):

```diff title="matches.py"
- from pydantic import BaseModel, Field, model_validator
+ from pydantic import BaseModel, Field, computed_field, field_validator, model_validator
```

Then define the model:

```diff title="matches.py"
+ class PaginationParams(BaseModel):
+     page: int = Field(1, ge=1, description="Page number (1-indexed)")
+     limit: int = Field(20, ge=1, le=100, description="Items per page (max 100)")
+
+     @computed_field
+     @property
+     def offset(self) -> int:
+         return (self.page - 1) * self.limit
```

`Field(1, ge=1, ...)` — the first positional argument is the default value, then constraints. No `model_validator` needed here — there's no relationship between `page` and `limit` that requires a cross-field check. But `offset` is a value we'll compute from them in the function body anyway, so it makes sense to pre-calculate it here and access it as `pagination.offset`.

## Step 6 — Create the SortParams Model

`sort_by` and `sort_order` control how results are ordered, not which results to include. They're a natural pair, separate from filtering.

```diff title="matches.py"
+ class SortParams(BaseModel):
+     sort_by: Optional[Literal["date", "sport", "status"]] = Field(
+         None,
+         title="Sort By",
+         description="Field to sort results by",
+     )
+     sort_order: Literal["asc", "desc"] = Field(
+         "asc",
+         title="Sort Order",
+         description="Sort direction — asc for ascending, desc for descending",
+     )
```

`Literal["date", "sport", "status"]` is a type that only accepts those exact string values — anything else is rejected automatically. This is the right tool when you have a small, fixed set of valid options that don't warrant a full `Enum` class. `sort_order` works the same way — only `"asc"` or `"desc"` are accepted, with `"asc"` as the default.

No `model_validator` is needed here because `sort_by` and `sort_order` don't have any conflict rules between them — any combination is valid.

## What Else Can Go Inside a Pydantic Model

We used `model_validator` to check relationships between fields. That's one of several things you can put inside a Pydantic model beyond just field definitions.

### Field-level validators

Run on a single field's value after it's parsed. Useful for transformations like normalizing input — stripping whitespace, lowercasing, reformatting dates, and so on.

We use this in two places. In `MatchFilters`, we normalize the `team` search query so that a user sending `?team=  Arsenal ` gets matched correctly:

```python title="matches.py"
class MatchFilters(BaseModel):
    team: Optional[str] = Field(None, min_length=3, max_length=50)

    @field_validator('team')
    @classmethod
    def normalize_team(cls, value):
        if value is not None:
            return value.strip().lower()
        return value
```

`value.strip()` removes leading and trailing whitespace — `"  Arsenal "` becomes `"Arsenal"`. `.lower()` then converts it to lowercase — `"Arsenal"` becomes `"arsenal"`. Since the `Match` model already stores team names in lowercase, the comparison just works without any extra `.lower()` in the filtering logic.

In `Match`, we normalize how team names and venues are stored. This is the validator that makes case-insensitive search possible in the first place:

```python title="matches.py"
class Match(BaseModel):
    ...

    @field_validator('home_team', 'away_team')
    @classmethod
    def normalize_team_name(cls, value):
        return value.strip().lower()

    @field_validator('venue')
    @classmethod
    def normalize_venue(cls, value):
        if value is not None:
            return value.strip().lower()
        return value
```

Now even if data comes in with inconsistent casing — `"Arsenal"`, `"ARSENAL"`, `"arsenal"` — it's always stored the same way. The filtering code never needs to call `.lower()` because the data is already normalized.

<Callout type="info" title="Why normalize in Match instead of at query time?">
  You could call `.lower()` every time you compare team names in the filtering logic, but that means every comparison in your function body carries that assumption. If you ever forget one, you get a subtle bug. Normalizing in `Match` when data is stored means the data is always consistent — the filtering code can just do `team in m.home_team` and trust it.
</Callout>

### Multiple model validators

You can have more than one `@model_validator` in the same class — each checks a different cross-field concern:

```python title="matches.py"
class DateRangeFilters(BaseModel):
    from_date: Optional[date] = None
    to_date: Optional[date] = None

    @model_validator(mode='after')
    def validate_order(self):
        if self.from_date and self.to_date and self.from_date > self.to_date:
            raise ValueError("from_date must be before or equal to to_date")
        return self

    @model_validator(mode='after')
    def validate_not_future(self):
        if self.to_date and self.to_date > date.today():
            raise ValueError("to_date cannot be in the future")
        return self
```

Each validator handles one specific concern. Both run automatically, in the order they're defined.

### Computed fields

Fields whose value is derived from other fields — calculated once when the model is created. We already added `offset` to `PaginationParams` in Step 5. Here's what it looks like and why it's useful:

```python title="matches.py"
class PaginationParams(BaseModel):
    page: int = Field(1, ge=1)
    limit: int = Field(20, ge=1, le=100)

    @computed_field
    @property
    def offset(self) -> int:
        return (self.page - 1) * self.limit
```

Instead of writing `(pagination.page - 1) * pagination.limit` in the function body, you access `pagination.offset` — the calculation lives next to the fields it depends on, and the function body stays free of arithmetic.

### Model config

Class-level settings that change how the model behaves:

```python title="matches.py"
class MatchFilters(BaseModel):
    model_config = {"extra": "forbid"}  # reject any unknown query params

    sport: Optional[Sport] = None
    ...
```

By default Pydantic silently ignores unknown fields — `?sport=football&typo=oops` works fine, `typo` is just dropped. With `extra: "forbid"`, the request is rejected with a validation error. Useful for internal APIs where you want to catch typos immediately, but avoid it for public APIs where adding a new optional parameter would break old clients.

<Callout type="info" title="The Pattern to Remember">
  Pydantic models are not just containers for data. They're the right place for any logic that's specifically about validating or transforming that data. If a check only makes sense given what a model's fields represent — whether one field contradicts another, whether a value needs normalizing, whether a derived value should be pre-computed — it belongs in the model, not scattered across your endpoint functions.

  This keeps your endpoint functions focused on what they're supposed to do: filtering, querying, and returning results.
</Callout>
