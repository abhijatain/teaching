@app.get("/matches", tags=["Matches"])
def list_matches(
    filters: Annotated[MatchFilters, Query()],
    date_range: Annotated[DateRangeFilters, Query()],
    pagination: Annotated[PaginationParams, Query()],
):
    matches = list(INITIAL_DATA)

    if filters.sport:
        matches = [m for m in matches if m.sport == filters.sport]
    if filters.status:
        matches = [m for m in matches if m.status == filters.status]
    if filters.winner:
        matches = [m for m in matches if m.winner == filters.winner]
    if filters.team:
        team_lower = filters.team.lower()
        matches = [
            m
            for m in matches
            if team_lower in m.home_team.lower() or team_lower in m.away_team.lower()
        ]

    if date_range.from_date:
        matches = [m for m in matches if m.date >= date_range.from_date]
    if date_range.to_date:
        matches = [m for m in matches if m.date <= date_range.to_date]

    total = len(matches)
    start = (pagination.page - 1) * pagination.limit
    paginated = matches[start : start + pagination.limit]

    return {
        "total": total,
        "page": pagination.page,
        "limit": pagination.limit,
        "results": paginated,
    }

# Request Examples

Your API has great validation and clear error messages, but users still need to understand what data to send. In this lesson, we'll add examples to your API documentation that show users exactly what valid requests look like.

---

## The Problem: Unclear Expectations

Visit `http://localhost:8000/docs` and click on `POST /matches`. The auto-generated schema shows field types and constraints:

```json title="What the docs show now"
{
  "home_team": "string",
  "away_team": "string",
  "sport": "football",
  "date": "2026-02-27",
  "status": "upcoming"
}
```

This is technically correct, but unhelpful:
- What's a realistic team name?
- What sports are actually available?
- What combinations of fields make sense together?

Users have to guess or experiment until they get it right.

<Callout type="warn" title="Why Examples Matter">
  Good API documentation doesn't just tell users what types are required — it shows them realistic, working examples they can copy and modify. This reduces support requests, speeds up integration, and creates a better developer experience.
</Callout>

---

## Step 1 — Add Field-Level Examples

The simplest way to add examples is directly in `Field()` definitions using the `examples` parameter:

```diff title="matches.py"
  class Match(BaseModel):
      id: int
-     home_team: str = Field(min_length=2, max_length=50)
+     home_team: str = Field(
+         min_length=2,
+         max_length=50,
+         examples=["Arsenal", "Real Madrid"],
+     )
-     away_team: str = Field(min_length=2, max_length=50)
+     away_team: str = Field(
+         min_length=2,
+         max_length=50,
+         examples=["Chelsea", "Barcelona"],
+     )
```

The `examples` parameter takes a list of example values. When you open `/docs`, these examples appear in the schema and can be used to pre-fill the request body.

<Callout type="info" title="Why a List?">
  `examples` takes a list because you can provide multiple example values. FastAPI will typically show the first one, but some tools might cycle through them or show all of them. Providing 2-3 varied examples helps users understand the range of valid inputs.
</Callout>

Add examples to the remaining fields:


  Use both: field examples help users understand individual constraints, endpoint examples help them understand complete workflows.
</Callout>

## Writing Good Examples

Good examples are:

**1. Realistic**
```python
# ✅ Good - realistic team names
"home_team": "Arsenal"

# ❌ Bad - placeholder values
"home_team": "string"
```

**2. Demonstrate constraints**
```python
# ✅ Good - shows valid jersey number range
"number": 10

# ❌ Bad - doesn't show the constraint exists
"number": 1
```

**3. Show relationships**
```python
# ✅ Good - winner matches the score
{
    "score_home": 3,
    "score_away": 1,
    "winner": "home_team",
    "status": "completed"
}

# ❌ Bad - inconsistent state
{
    "score_home": 3,
    "score_away": 1,
    "winner": "away_team",  # Doesn't match score!
    "status": "completed"
}
```

**4. Cover common use cases**
```python
openapi_examples={
    "basic": {...},       # Minimal required fields
    "complete": {...},    # All optional fields included
    "edge_case": {...},   # Unusual but valid scenario
}
```

<Callout type="warn" title="Examples Are Part of Your API Contract">
  Examples show users what your API expects. If your examples are wrong or inconsistent with validation rules, users will get confused and frustrated. Keep examples updated when you change validation rules.
</Callout>

## Where Examples Appear

Examples you add show up in multiple places:

**1. Interactive API docs** (`/docs`)
- Pre-fill the "Try it out" form
- Show in dropdown for `openapi_examples`
- Display in schema section

**2. OpenAPI JSON** (`/openapi.json`)
- Part of the schema definition
- Used by API client generators
- Read by API testing tools

**3. Alternative doc UIs** (`/redoc`)
- Show in request body schema
- Display alongside field descriptions

**4. Client code generators**
- Some tools use examples to generate default values
- Help developers understand expected formats

<Callout type="info" title="Examples Improve the Developer Experience">
  Good examples reduce:
  - Time to first successful API call
  - Support questions about request format
  - Trial and error during integration
  - Documentation that just repeats the schema
</Callout>
