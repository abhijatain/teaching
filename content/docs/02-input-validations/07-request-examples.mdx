# Request Examples

Your API is fully validated and supports nested models. But when developers visit your API documentation, they need to know what valid requests look like. In this lesson, we'll add examples to your models and endpoints to make your API self-documenting.

---

## Where We Are

Visit `http://localhost:8000/docs` and click on `POST /matches`. You'll see the request body schema, but the example values are generic:

```json title="Current auto-generated example"
{
  "id": 0,
  "home_team": "string",
  "away_team": "string",
  "venue": "string",
  "date": "2026-02-27",
  "sport": "football",
  "status": "upcoming",
  "winner": "home_team",
  "home_lineup": [
    {
      "number": 0,
      "name": "string",
      "position": "goalkeeper",
      "is_captain": true
    }
  ]
}
```

This works, but it doesn't help developers understand what realistic data looks like. `"home_team": "string"` is technically correct, but `"home_team": "Arsenal"` is much more informative.

<Callout type="info" title="Why Examples Matter">
  Good examples reduce integration time. Developers can copy a working example, modify it for their needs, and get a successful response faster. Poor examples force developers to read documentation, experiment with values, and deal with validation errors — all of which slows down adoption.
</Callout>

<Callout type="info" title="Examples vs Defaults">
  Don't confuse `examples` with default values:
  - `Field(default="Arsenal")` — every match would have Arsenal as home team unless specified
  - `Field(examples=["Arsenal"])` — Arsenal is shown as a sample value, but nothing is filled in automatically
  
  Use defaults for fields that genuinely have a common default. Use examples for fields where the user needs to provide a value but you want to show them what valid values look like.
</Callout>

</Callout>

## Writing Good Examples

Good examples are:

**1. Realistic**
```python
# ✅ Good - realistic team names
"home_team": "Arsenal"

# ❌ Bad - placeholder values
"home_team": "string"
```

**2. Demonstrate constraints**
```python
# ✅ Good - shows valid jersey number range
"number": 10

# ❌ Bad - doesn't show the constraint exists
"number": 1
```

**3. Show relationships**
```python
# ✅ Good - completed match with winner
{
    "status": "completed",
    "winner": "home_team"
}

# ❌ Bad - upcoming match shouldn't have a winner
{
    "status": "upcoming",
    "winner": "home_team"
}
```

**4. Cover common use cases**
```python
openapi_examples={
    "basic": {...},       # Minimal required fields
    "complete": {...},    # All optional fields included
    "with_lineup": {...}, # Shows nested structures
}
```

<Callout type="warn" title="Examples Are Part of Your API Contract">
  Examples show users what your API expects. If your examples are wrong or inconsistent with validation rules, users will get confused and frustrated. Keep examples updated when you change validation rules.
</Callout>

## Where Examples Appear

Examples you add show up in multiple places:

**1. Interactive API docs** (`/docs`)
- Pre-fill the "Try it out" form
- Show in dropdown for `openapi_examples`
- Display in schema section

**2. OpenAPI JSON** (`/openapi.json`)
- Part of the schema definition
- Used by API client generators
- Read by API testing tools

**3. Alternative doc UIs** (`/redoc`)
- Show in request body schema
- Display alongside field descriptions

**4. Client code generators**
- Some tools use examples to generate default values
- Help developers understand expected formats
