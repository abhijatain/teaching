# Nested Models and Complex Data

Our matches have basic info (teams, scores, dates), but real sports data is richer: lineups, events, player stats. Let's model complex, nested data structures.

## The Problem: Flat Data

Currently, our `Match` model is flat:

```python title="matches.py - Current model"
class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    venue: Optional[str] = None
    score_home: Optional[int] = None
    score_away: Optional[int] = None
```

But real matches have:
- **Players** — with names, numbers, positions
- **Events** — goals, cards, substitutions with timestamps
- **Statistics** — possession, shots, fouls
- **Lineups** — starting XI and substitutes

We need **nested models** — models inside models.

<Callout type="warn" title="Why Flat Models Don't Scale">
  Flat models force you to:
  - Add endless optional fields (`goal_1_scorer`, `goal_2_scorer`, `goal_3_scorer`...)
  - Store structured data as JSON strings (loses validation)
  - Make separate API calls for related data (N+1 queries)
  
  Nested models keep related data together and properly validated.
</Callout>

## List Fields with Type Parameters

Let's start simple: a match can have multiple tags (like "rivalry", "playoff", "friendly"):

```python title="matches.py"
class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    tags: list[str] = Field(default_factory=list)  # !mark
```

<Callout type="info" title="Why default_factory=list?">
  Don't use `tags: list[str] = []` — this creates a **single shared list** across all Match instances (Python gotcha). Use `default_factory=list` to create a **new list** for each instance.
  
  ```python
  # ❌ Wrong - shared list
  tags: list[str] = []
  
  # ✅ Right - new list per instance
  tags: list[str] = Field(default_factory=list)
  ```
</Callout>

Now you can create matches with tags:

```json title="Match with tags"
{
  "id": 1,
  "home_team": "Arsenal",
  "away_team": "Chelsea",
  "sport": "football",
  "date": "2025-10-21",
  "status": "completed",
  "tags": ["rivalry", "london-derby", "high-stakes"]
}
```

## Sets for Unique Values

Tags shouldn't repeat. Use a `set` to enforce uniqueness:

```python title="matches.py"
class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    tags: set[str] = Field(default_factory=set)  # !mark
```

Now if a user sends duplicate tags:

```json title="Request with duplicates"
{
  "tags": ["rivalry", "playoff", "rivalry", "playoff"]
}
```

Pydantic automatically deduplicates:

```json title="Stored as"
{
  "tags": ["rivalry", "playoff"]
}
```

<Callout type="info" title="List vs Set">
  - **`list[str]`** — ordered, allows duplicates ("goal", "goal", "goal")
  - **`set[str]`** — unordered, unique values ("home-win", "rivalry")
  
  Use sets for categories, tags, flags. Use lists for ordered sequences.
</Callout>

## Nested Models: Player Model

Let's create a `Player` model and nest it inside `Match`:

```python title="matches.py"
class Player(BaseModel):  # !mark
    """Individual player in a match"""
    number: int = Field(ge=1, le=99, description="Jersey number")  # !mark
    name: str = Field(min_length=2, max_length=50)  # !mark
    position: Literal["goalkeeper", "defender", "midfielder", "forward"]  # !mark
    is_captain: bool = Field(default=False)  # !mark


class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    home_lineup: Optional[list[Player]] = None  # !mark
    away_lineup: Optional[list[Player]] = None  # !mark
```

Now you can create a match with full lineups:

```json title="Match with nested players"
{
  "id": 1,
  "home_team": "Arsenal",
  "away_team": "Chelsea",
  "sport": "football",
  "date": "2025-10-21",
  "status": "completed",
  "home_lineup": [
    {
      "number": 1,
      "name": "David Raya",
      "position": "goalkeeper",
      "is_captain": false
    },
    {
      "number": 10,
      "name": "Martin Ødegaard",
      "position": "midfielder",
      "is_captain": true
    }
  ],
  "away_lineup": [
    {
      "number": 7,
      "name": "Raheem Sterling",
      "position": "forward",
      "is_captain": false
    }
  ]
}
```

<Callout type="info" title="Automatic Nested Validation">
  Pydantic validates **every player** in the lineup:
  - Jersey number must be 1-99
  - Name must be 2-50 characters
  - Position must be one of the four allowed values
  
  If any player is invalid, the entire request is rejected with a clear error showing which player and which field failed.
</Callout>

## Match Events

Let's model events that happen during a match (goals, cards, substitutions):

```python title="matches.py"
class MatchEvent(BaseModel):  # !mark
    """An event that occurred during the match"""
    minute: int = Field(ge=0, le=120, description="Minute when event occurred")  # !mark
    event_type: Literal["goal", "yellow_card", "red_card", "substitution"]  # !mark
    team: Literal["home", "away"]  # !mark
    player_name: str = Field(min_length=2, max_length=50)  # !mark
    description: Optional[str] = Field(None, max_length=200)  # !mark


class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    home_lineup: Optional[list[Player]] = None
    away_lineup: Optional[list[Player]] = None

```

Create a match with events:

```json title="Match with timeline"

  "away_team": "Chelsea",
  "sport": "football",

  "events": [
    {
      "minute": 23,

      "event_type": "yellow_card",
      "team": "away",
      "player_name": "Enzo Fernández",
```

## Deeply Nested Models

You can nest models arbitrarily deep. Let's create match statistics with multiple levels:

```python title="matches.py"
class TeamStats(BaseModel):  # !mark
    """Statistics for one team"""
    possession_percent: int = Field(ge=0, le=100)  # !mark
    shots: int = Field(ge=0)  # !mark
    shots_on_target: int = Field(ge=0)  # !mark
    corners: int = Field(ge=0)  # !mark
    fouls: int = Field(ge=0)  # !mark
    yellow_cards: int = Field(ge=0)  # !mark
    red_cards: int = Field(ge=0)  # !mark


class MatchStatistics(BaseModel):  # !mark
    """Complete match statistics"""
    home_stats: TeamStats  # !mark
    away_stats: TeamStats  # !mark
    attendance: Optional[int] = Field(None, ge=0, description="Number of spectators")  # !mark
    referee: Optional[str] = Field(None, min_length=2, max_length=50)  # !mark


class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    home_lineup: Optional[list[Player]] = None
    away_lineup: Optional[list[Player]] = None
    events: list[MatchEvent] = Field(default_factory=list)
    statistics: Optional[MatchStatistics] = None  # !mark
```

Three levels of nesting:
1. `Match` contains `MatchStatistics`
2. `MatchStatistics` contains `TeamStats` (twice)
3. `TeamStats` has individual stat fields

Example:

```json title="Deeply nested match data"
{
  "id": 1,
  "home_team": "Arsenal",
  "away_team": "Chelsea",
  "sport": "football",
  "date": "2025-10-21",
  "status": "completed",
  "statistics": {
    "home_stats": {
      "possession_percent": 58,
      "shots": 15,
      "shots_on_target": 7,
      "corners": 8,
      "fouls": 12,
      "yellow_cards": 2,
      "red_cards": 0
    },
    "away_stats": {
      "possession_percent": 42,
      "shots": 9,
      "shots_on_target": 4,
      "corners": 5,
      "fouls": 15,
      "yellow_cards": 3,
      "red_cards": 1
    },
    "attendance": 60000,
    "referee": "Michael Oliver"
  }
}
```

<Callout type="info" title="Validation at Every Level">
  Pydantic validates the entire tree:
  - Possession percentages are 0-100
  - All stats are non-negative
  - Referee name is 2-50 characters
  - Attendance is non-negative
  
  One validation rule, recursively applied throughout the structure.
</Callout>

## Special Pydantic Types

Pydantic provides special types for common use cases. Let's use `HttpUrl` for official match pages:

```python title="matches.py"
from pydantic import HttpUrl  # !mark


class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    official_page: Optional[HttpUrl] = Field(  # !mark
        None,
        description="Official match page URL",
        examples=["https://www.premierleague.com/match/12345"],
    )
```

Try creating a match with an invalid URL:

```json title="❌ Invalid URL"
{
  "official_page": "not-a-url"
}
```

Error: "invalid or missing URL scheme"

```json title="✅ Valid URL"
{
  "official_page": "https://www.premierleague.com/match/12345"
}
```

<Callout type="info" title="Other Special Types">
  Pydantic provides many specialized types:
  - **`HttpUrl`** — valid HTTP/HTTPS URL
  - **`EmailStr`** — valid email address
  - **`UUID`** — valid UUID
  - **`IPvAnyAddress`** — valid IP address
  - **`PositiveInt`** — int > 0
  - **`constr()`** — constrained string
  
  See [Pydantic's Type Overview](https://docs.pydantic.dev/latest/api/types/) for the full list.
</Callout>

## Lists of Pure Data

Sometimes your endpoint accepts a list as the top-level structure. For example, creating multiple match events at once:

```python title="matches.py"
@app.post("/matches/{match_id}/events/batch", tags=["Matches"])
def add_multiple_events(
    match_id: Annotated[int, Path(ge=1)],
    events: list[MatchEvent],  # !mark (list at top level)
):
    """Add multiple events to a match in one request"""
    for match in INITIAL_DATA:
        if match.id == match_id:
            match.events.extend(events)
            return {"added": len(events), "total_events": len(match.events)}
    
    return {"error": "Match not found"}
```

Request body is a JSON array:

```json title="Top-level array"
[
  {
    "minute": 12,
    "event_type": "goal",
    "team": "home",
    "player_name": "Gabriel Jesus"
  },
  {
    "minute": 34,
    "event_type": "yellow_card",
    "team": "away",
    "player_name": "Enzo Fernández"
  }
]
```

<Callout type="warn" title="Array vs Object">
  Top-level arrays are less flexible:
  - Can't add metadata (like `"timestamp": "2025-10-21T15:30:00"`)
  - Harder to version (breaking change to wrap in object later)
  
  Generally prefer wrapping in an object:
  ```json
  {
    "events": [...],
    "timestamp": "2025-10-21T15:30:00"
  }
  ```
</Callout>

## Arbitrary Dictionaries

Sometimes you need flexible key-value data. For example, storing custom metadata:

```python title="matches.py"
class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    sport: Sport
    date: date
    status: Status
    metadata: dict[str, str] = Field(default_factory=dict)  # !mark
```

Users can store anything:

```json title="Custom metadata"
{
  "id": 1,
  "home_team": "Arsenal",
  "away_team": "Chelsea",
  "metadata": {
    "competition": "Premier League",
    "season": "2025-26",
    "matchday": "10",
    "broadcast": "Sky Sports"
  }
}
```

<Callout type="warn" title="When to Use Dicts">
  Use `dict[str, str]` when:
  - Keys are truly dynamic (user-defined, plugin-based)
  - You don't know the structure ahead of time
  
  Don't use dicts when:
  - You know the fields (use a Pydantic model instead)
  - You need validation (dicts don't validate contents)
  
  Dicts sacrifice validation for flexibility — use them sparingly.
</Callout>

## Typed Dictionaries

If you need dictionaries with validated values, specify the value type:

```python title="matches.py"
class Match(BaseModel):
    id: int
    home_team: str
    away_team: str
    player_ratings: dict[str, float] = Field(  # !mark
        default_factory=dict,
        description="Player name → rating (0.0-10.0)",
    )
```

Example:

```json title="Player ratings"
{
  "player_ratings": {
    "Gabriel Jesus": 8.5,
    "Martin Ødegaard": 9.0,
    "David Raya": 7.5
  }
}
```

Pydantic validates all values are floats. Try sending:

```json title="❌ Invalid value type"
{
  "player_ratings": {
    "Gabriel Jesus": "excellent"
  }
}
```

Error: "value is not a valid float"

<Callout type="info" title="Dict Type Parameters">
  ```python
  dict[str, int]      # Keys: str, Values: int
  dict[int, float]    # Keys: int, Values: float
  dict[str, Player]   # Keys: str, Values: Player model
  ```
  
  Note: JSON only supports string keys, but Pydantic converts numeric strings automatically.
</Callout>

## Complete Rich Match Model

Here's a full model with all nested structures:

```python title="matches.py"
from pydantic import BaseModel, Field, HttpUrl, model_validator
from typing import Optional, Literal
from datetime import date


class Player(BaseModel):
    """Player in a lineup"""
    number: int = Field(ge=1, le=99)
    name: str = Field(min_length=2, max_length=50)
    position: Literal["goalkeeper", "defender", "midfielder", "forward"]
    is_captain: bool = False


class MatchEvent(BaseModel):
    """Event during the match"""
    minute: int = Field(ge=0, le=120)
    event_type: Literal["goal", "yellow_card", "red_card", "substitution"]
    team: Literal["home", "away"]
    player_name: str = Field(min_length=2, max_length=50)
    description: Optional[str] = Field(None, max_length=200)


class TeamStats(BaseModel):
    """Statistics for one team"""
    possession_percent: int = Field(ge=0, le=100)
    shots: int = Field(ge=0)
    shots_on_target: int = Field(ge=0)
    corners: int = Field(ge=0)
    fouls: int = Field(ge=0)
    yellow_cards: int = Field(ge=0)
    red_cards: int = Field(ge=0)
    
    @model_validator(mode='after')
    def validate_shots(self):
        """Shots on target can't exceed total shots"""
        if self.shots_on_target > self.shots:
            raise ValueError("shots_on_target cannot exceed total shots")
        return self


class MatchStatistics(BaseModel):
    """Complete match statistics"""
    home_stats: TeamStats
    away_stats: TeamStats
    attendance: Optional[int] = Field(None, ge=0)
    referee: Optional[str] = Field(None, min_length=2, max_length=50)


class Match(BaseModel):
    """Complete match data with all nested information"""
    id: int
    home_team: str = Field(min_length=2, max_length=50)
    away_team: str = Field(min_length=2, max_length=50)
    sport: Literal["football", "cricket", "hockey", "basketball", "baseball"]
    date: date
    status: Literal["upcoming", "completed", "cancelled", "abandoned"]
    venue: Optional[str] = Field(None, min_length=3, max_length=100)
    
    # Scores
    score_home: Optional[int] = Field(None, ge=0)
    score_away: Optional[int] = Field(None, ge=0)
    
    # Lineups
    home_lineup: Optional[list[Player]] = None
    away_lineup: Optional[list[Player]] = None
    
    # Events timeline
    events: list[MatchEvent] = Field(default_factory=list)
    
    # Statistics
    statistics: Optional[MatchStatistics] = None
    
    # Metadata
    tags: set[str] = Field(default_factory=set)
    official_page: Optional[HttpUrl] = None
    player_ratings: dict[str, float] = Field(default_factory=dict)
    
    @model_validator(mode='after')
    def validate_teams_different(self):
        """Ensure home and away teams are different"""
        if self.home_team == self.away_team:
            raise ValueError("home_team and away_team must be different")
        return self


# Create endpoint
@app.post("/matches/detailed", tags=["Matches"])
def create_detailed_match(match: Match):
    """Create a match with full details including lineups, events, and statistics"""
    # Generate new ID
    new_id = INITIAL_DATA[-1].id + 1 if INITIAL_DATA else 1
    match.id = new_id
    
    INITIAL_DATA.append(match)
    return match
```

## Editor Support Benefits

One huge advantage of nested models: **editor autocomplete at every level**.

```python title="Type-safe navigation"
match = Match(...)

# Autocomplete shows all Match fields
match.home_team
match.statistics

# Autocomplete shows all MatchStatistics fields
match.statistics.home_stats
match.statistics.referee

# Autocomplete shows all TeamStats fields
match.statistics.home_stats.possession_percent
match.statistics.home_stats.shots_on_target

# Type checking works too
match.statistics.home_stats.possession_percent = 150  # ❌ Error: must be ≤ 100
```

With plain dictionaries, you get none of this:

```python title="No type safety with dicts"
match = {"statistics": {"home_stats": {"possession": 58}}}

match["statistics"]["home_stats"]["possesion"]  # ❌ Typo - no error until runtime!
```

<Callout type="info" title="Type Safety Everywhere">
  Nested Pydantic models give you:
  - Autocomplete in editors (VS Code, PyCharm)
  - Type checking with mypy/pyright
  - Refactoring support (rename fields safely)
  - Documentation in docstrings
  
  This is **massive** for large codebases — you catch errors before running the code.
</Callout>

## Testing Nested Models

Nested models are easy to test because they're composable:

```python title="test_matches.py"
def test_team_stats_validation():
    """Test TeamStats validates shots correctly"""
    with pytest.raises(ValueError, match="shots_on_target cannot exceed total shots"):
        TeamStats(
            possession_percent=50,
            shots=10,
            shots_on_target=15,  # More on target than total!
            corners=5,
            fouls=10,
            yellow_cards=2,
            red_cards=0,
        )


def test_valid_team_stats():
    """Test valid TeamStats"""
    stats = TeamStats(
        possession_percent=58,
        shots=15,
        shots_on_target=7,
        corners=8,
        fouls=12,
        yellow_cards=2,
        red_cards=0,
    )
    assert stats.possession_percent == 58
    assert stats.shots_on_target <= stats.shots


def test_match_with_statistics():
    """Test creating a match with full statistics"""
    match = Match(
        id=1,
        home_team="Arsenal",
        away_team="Chelsea",
        sport="football",
        date=date(2025, 10, 21),
        status="completed",
        statistics=MatchStatistics(
            home_stats=TeamStats(
                possession_percent=60,
                shots=15,
                shots_on_target=7,
                corners=8,
                fouls=12,
                yellow_cards=2,
                red_cards=0,
            ),
            away_stats=TeamStats(
                possession_percent=40,
                shots=9,
                shots_on_target=4,
                corners=5,
                fouls=15,
                yellow_cards=3,
                red_cards=1,
            ),
            attendance=60000,
            referee="Michael Oliver",
        ),
    )
    assert match.statistics.home_stats.possession_percent == 60
    assert match.statistics.attendance == 60000
```

## When to Use Nested Models

✅ **Use nested models when:**
- Data has clear hierarchical structure (match → lineups → players)
- Related data is always fetched together
- You need validation at multiple levels
- You want type safety and autocomplete

❌ **Don't use nested models when:**
- Data is frequently accessed separately (make separate endpoints)
- Nesting is too deep (>4 levels gets unwieldy)
- Performance matters (deeply nested validation is slower)
- Data structure is truly dynamic (use dicts)

## Summary: Nested Models

```python title="Complexity levels"
# Level 1: Simple fields
name: str

# Level 2: List of simple values
tags: list[str]

# Level 3: List of models
players: list[Player]

# Level 4: Model containing model containing lists
statistics: MatchStatistics  # contains TeamStats with list fields

# Level 5: Arbitrary depth
# You can nest as deep as needed, but 3-4 levels is usually the practical limit
```

<Callout type="info" title="Key Takeaway">
  Nested models give you:
  - **Structure** — organize complex data logically
  - **Validation** — recursive validation at all levels
  - **Type safety** — autocomplete and error checking
  - **Documentation** — self-documenting hierarchical schemas
  
  All while keeping your code clean and Pythonic.
</Callout>
