# Path Parameter Validation

We have a working `/matches/{match_id}` endpoint, but it accepts any integer. Let's add validation to make it more robust.

## The Current Endpoint

Here's our get-by-ID route:

```python title="matches.py"
@app.get("/matches/{match_id}", tags=["Matches"])
def get_match_by_id(match_id: int):
    for match in INITIAL_DATA:
        if match.id == match_id:
            return match
    return {"error": "Match not found"}
```

<Callout type="info" title="What We Have">
  FastAPI automatically:
  - Extracts `match_id` from the URL path
  - Converts it to an integer
  - Rejects non-numeric values like `/matches/abc`
  
  This is basic type validation — but we can do better.
</Callout>

---

## The Problem: Negative IDs

Try these requests:

```bash title="Valid integer, but nonsense ID"
curl http://localhost:8000/matches/-1
curl http://localhost:8000/matches/0
curl http://localhost:8000/matches/-999999
```

All these pass validation and reach your function. Your code loops through `INITIAL_DATA`, finds nothing, and returns "Match not found."

<Callout type="warn" title="Why Is This Bad?">
  IDs should always be positive. Accepting negative or zero IDs means:
  - Wasted processing time searching for impossible values
  - Unclear API semantics (is `-1` special? Is `0` valid?)
  - Potential bugs if you later add special meaning to negative IDs
  
  Better to reject them immediately with a clear validation error.
</Callout>

---

## The Solution: Path Parameter Validation

Just like `Query` validates query parameters, `Path` validates path parameters. Import it:

```diff title="matches.py"
  from datetime import date
  from enum import Enum
  from typing import List, Optional
  from typing import Annotated

- from fastapi import FastAPI, Query
+ from fastapi import FastAPI, Path, Query
  from pydantic import BaseModel

  # ... rest of code ...
```

## Adding a Minimum Value

Use `ge=1` (greater than or equal to 1) to require positive IDs:

```python title="matches.py"
@app.get("/matches/{match_id}", tags=["Matches"])
def get_match_by_id(
    match_id: Annotated[int, Path(ge=1)],  # !mark
):
    for match in INITIAL_DATA:
        if match.id == match_id:
            return match
    return {"error": "Match not found"}
```

### Understanding the Syntax

```python title="Breaking it down"
match_id: Annotated[int, Path(ge=1)]
          ^^^^^^^^^
          Annotated wraps the type

match_id: Annotated[int, Path(ge=1)]
                    ^^^
                    Base type is int

match_id: Annotated[int, Path(ge=1)]
                         ^^^^^^^^^^
                         Validation: greater than or equal to 1
```

Now try the invalid requests:

```bash title="Negative ID rejected"
curl http://localhost:8000/matches/-1
```

Response:

```json title="Validation error"
{
  "detail": [
    {
      "loc": ["path", "match_id"],
      "msg": "ensure this value is greater than or equal to 1",
      "type": "value_error.number.not_ge",
      "ctx": {"limit_value": 1}
    }
  ]
}
```

<Callout type="info" title="Validation Before Execution">
  FastAPI intercepts the request **before your function runs**. Invalid IDs are rejected immediately. Your code never sees `-1` or `0` — it only processes valid positive integers.
</Callout>

## Numeric Validation Options

Path parameters support four numeric constraints:

| Constraint | Meaning | Example |
|-----------|---------|---------|
| `ge=n` | Greater than or equal | `ge=1` (ID must be ≥ 1) |
| `gt=n` | Greater than | `gt=0` (ID must be > 0) |
| `le=n` | Less than or equal | `le=1000` (ID must be ≤ 1000) |
| `lt=n` | Less than | `lt=1000` (ID must be < 1000) |

<Callout type="info" title="ge vs gt">
  What's the difference?
  - `ge=1` accepts: 1, 2, 3, 4...
  - `gt=0` accepts: 1, 2, 3, 4... (same result)
  
  But with decimals:
  - `ge=1.0` accepts: 1.0, 1.5, 2.0...
  - `gt=0.9` accepts: 0.91, 1.0, 1.5... (includes values between 0.9 and 1.0)
  
  For IDs, `ge=1` is clearer — it explicitly means "1 or higher."
</Callout>

## Adding an Upper Bound

What if your system can only handle IDs up to a certain size? Add a maximum:

```python title="matches.py"
@app.get("/matches/{match_id}", tags=["Matches"])
def get_match_by_id(
    match_id: Annotated[int, Path(ge=1, le=999999)],  # !mark
):
    for match in INITIAL_DATA:
        if match.id == match_id:
            return match
    return {"error": "Match not found"}
```

Now IDs must be between 1 and 999,999 inclusive. Try:

```bash title="Out of range"
curl http://localhost:8000/matches/1000000
```

Response:

```json title="Validation error"
{
  "detail": [
    {
      "loc": ["path", "match_id"],
      "msg": "ensure this value is less than or equal to 999999",
      "type": "value_error.number.not_le",
      "ctx": {"limit_value": 999999}
    }
  ]
}
```

<Callout type="warn" title="Choose Realistic Limits">
  Don't set arbitrary limits. If your database uses 64-bit integers, it can handle IDs up to 9 quintillion. Setting `le=999999` only makes sense if you have a real reason — like migrating from a legacy system with 6-digit IDs, or intentionally keeping a test environment small.
  
  For most production APIs, just use `ge=1` and let the database handle the upper limit naturally.
</Callout>

## Adding Documentation

Just like with `Query`, you can add metadata that appears in the auto-generated docs:

```python title="matches.py"
@app.get("/matches/{match_id}", tags=["Matches"])
def get_match_by_id(
    match_id: Annotated[
        int,
        Path(
            ge=1,
            title="Match ID",  # !mark
            description="The unique identifier of the match to retrieve",  # !mark
        ),
    ],
):
    for match in INITIAL_DATA:
        if match.id == match_id:
            return match
    return {"error": "Match not found"}
```

Visit `http://localhost:8000/docs` and look at the `/matches/{match_id}` endpoint. The parameter now has a title and description visible in the UI.

<Callout type="info" title="Path Parameters Are Always Required">
  Unlike query parameters, path parameters **can never be optional**. The URL `/matches/` (without an ID) matches a different route entirely. So even if you write `= None` or set a default value, it has no effect — path parameters are always required.
  
  This is why you don't see `Optional[int]` with path parameters.
</Callout>

## Combining Path and Query Parameters

You can validate both path and query parameters in the same endpoint:

```python title="matches.py"
@app.get("/matches/{match_id}", tags=["Matches"])
def get_match_by_id(
    match_id: Annotated[
        int,
        Path(ge=1, title="Match ID"),
    ],
    include_details: Annotated[
        bool,
        Query(description="Include venue and other details"),
    ] = False,  # !mark
):
    for match in INITIAL_DATA:
        if match.id == match_id:
            if include_details:
                return match
            else:
                # Return minimal info
                return {
                    "id": match.id,
                    "home_team": match.home_team,
                    "away_team": match.away_team,
                    "date": match.date,
                }
    return {"error": "Match not found"}
```

Users can now call:

```bash
# Minimal info
curl http://localhost:8000/matches/1

# Full details
curl "http://localhost:8000/matches/1?include_details=true"
```

<Callout type="info" title="Path vs Query">
  - **Path parameters** (`/matches/{id}`) identify **which resource** you want
  - **Query parameters** (`?include_details=true`) modify **how you want it**
  
  Path = what. Query = how. This is RESTful design.
</Callout>

## Real Example: Pagination with Constraints

Here's a practical example combining multiple validated parameters:

```python title="matches.py"
@app.get("/matches", tags=["Matches"])
def list_matches(
    sport: Annotated[
        Optional[Sport],
        Query(description="Filter by sport"),
    ] = None,
    page: Annotated[
        int,
        Query(ge=1, title="Page Number", description="Page of results (1-indexed)"),
    ] = 1,  # !mark
    limit: Annotated[
        int,
        Query(ge=1, le=100, title="Results Per Page", description="Max 100 items"),
    ] = 20,  # !mark
):
    results = INITIAL_DATA

    # Apply sport filter
    if sport:
        results = [m for m in results if m.sport == sport]

    # Calculate pagination
    start = (page - 1) * limit  # !mark
    end = start + limit  # !mark
    paginated = results[start:end]  # !mark

    return {
        "page": page,
        "limit": limit,
        "total": len(results),
        "results": paginated,
    }
```

### Understanding Pagination Math

```python title="How pagination works"
# User wants page 2 with limit 3:
page = 2
limit = 3

start = (2 - 1) * 3 = 3    # Start at index 3 (skip first 3 items)
end = 3 + 3 = 6            # End at index 6 (take 3 items)

results[3:6]               # Gets items at indices 3, 4, 5
```

Try it:

```bash title="First page"
curl "http://localhost:8000/matches?page=1&limit=3"
```

```bash title="Invalid: page 0"
curl "http://localhost:8000/matches?page=0&limit=3"
```

```bash title="Invalid: limit too high"
curl "http://localhost:8000/matches?page=1&limit=200"
```

Both invalid requests are rejected with clear validation errors — before any database query happens.

<Callout type="warn" title="Why Limit Pagination Size?">
  Without `le=100`, users could request:
  ```
  /matches?limit=999999999
  ```
  
  This could:
  - Exhaust your server's memory
  - Cause timeouts
  - Be a denial-of-service attack vector
  
  Always cap pagination limits to something reasonable (10-100 items per page).
</Callout>

## Float Validations

Path and query parameters can also validate floats:

```python title="matches.py - Hypothetical example"
@app.get("/matches/by-rating/{min_rating}", tags=["Matches"])
def get_matches_by_rating(
    min_rating: Annotated[
        float,
        Path(gt=0.0, le=5.0, title="Minimum Rating"),  # !mark
    ],
):
    # Filter matches by some rating system
    # min_rating must be: 0 < rating ≤ 5.0
    return {"message": f"Finding matches with rating > {min_rating}"}
```

Try:

```bash
curl http://localhost:8000/matches/by-rating/4.5   # ✅ Valid
curl http://localhost:8000/matches/by-rating/0.0   # ❌ Must be > 0
curl http://localhost:8000/matches/by-rating/6.0   # ❌ Must be ≤ 5.0
```

<Callout type="info" title="Why gt=0.0 instead of ge=0.0?">
  `gt=0.0` means "strictly greater than zero" — so 0.0 is rejected but 0.01 is accepted. This is useful when zero has a special meaning (like "no rating" or "undefined"). If you want to allow exactly 0.0, use `ge=0.0` instead.
</Callout>

## Parameter Order (Old Style Caveat)

<Callout type="info" title="Annotated Makes This Irrelevant">
  If you're using `Annotated` (which you should), you can **skip this section**. Parameter order doesn't matter. This is only relevant for the old pre-0.95.0 style.
</Callout>

In the old style (without `Annotated`), Python's syntax rules caused issues:

```python title="❌ Old style - syntax error"
@app.get("/matches/{match_id}")
def get_match_by_id(
    match_id: int = Path(ge=1),  # Has default value (the Path object)
    required_param: str,          # No default value
):
    pass
```

Python complains: "non-default argument follows default argument."

The old workaround was using `*` to force keyword-only arguments:

```python title="Old style workaround"
@app.get("/matches/{match_id}")
def get_match_by_id(
    *,                           # Force all args to be keyword-only
    match_id: int = Path(ge=1),
    required_param: str,
):
    pass
```

With `Annotated`, this problem doesn't exist:

```python title="✅ New style - no issues"
@app.get("/matches/{match_id}")
def get_match_by_id(
    match_id: Annotated[int, Path(ge=1)],  # !mark
    required_param: str,
):
    pass
```

Order doesn't matter because you're not using the function's default value mechanism — `Path()` is just metadata inside `Annotated`.

## Complete Example: Validated Match Endpoint

Here's a fully validated endpoint combining everything we've learned:

```python title="matches.py"
@app.get("/matches/{match_id}", tags=["Matches"])
def get_match_by_id(
    match_id: Annotated[
        int,
        Path(
            ge=1,
            title="Match ID",
            description="The unique identifier of the match (must be positive)",
        ),
    ],
    include_venue: Annotated[
        bool,
        Query(
            title="Include Venue",
            description="Whether to include venue information in the response",
        ),
    ] = True,
    include_details: Annotated[
        bool,
        Query(
            title="Include Details",
            description="Include all match metadata (date, status, winner)",
        ),
    ] = True,
):
    # Find the match
    for match in INITIAL_DATA:
        if match.id == match_id:
            # Build response based on query parameters
            response = {
                "id": match.id,
                "home_team": match.home_team,
                "away_team": match.away_team,
            }
            
            if include_details:
                response.update({
                    "sport": match.sport,
                    "date": match.date,
                    "status": match.status,
                    "winner": match.winner,
                })
            
            if include_venue and match.venue:
                response["venue"] = match.venue
            
            return response
    
    return {"error": "Match not found"}
```

This endpoint:
- ✅ Validates `match_id` is a positive integer
- ✅ Provides clear documentation for all parameters
- ✅ Allows flexible response formats via query parameters
- ✅ Rejects invalid input before processing

## What Path Can't Validate

Just like `Query`, `Path` handles individual parameter validation but not **relationships between parameters**:

```python title="matches.py - Hypothetical date range endpoint"
@app.get("/matches/range/{start_year}/{end_year}", tags=["Matches"])
def get_matches_by_year_range(
    start_year: Annotated[int, Path(ge=1900, le=2100)],
    end_year: Annotated[int, Path(ge=1900, le=2100)],
):
    # Both years are valid individually, but is start_year ≤ end_year?
    if start_year > end_year:  # !mark
        return {"error": "start_year must be before or equal to end_year"}  # !mark
    
    # Filter matches by year range...
    return {"message": f"Matches from {start_year} to {end_year}"}
```

Try:

```bash
curl http://localhost:8000/matches/range/2025/2020  # Backwards range
```

The individual values (2025 and 2020) are valid years, but their relationship is wrong. You have to check this manually in your code.

<Callout type="warn" title="Two Validation Layers">
  - **FastAPI/Pydantic:** Validates individual parameter types, formats, and ranges
  - **Your Business Logic:** Validates relationships between parameters
  
  Both are necessary for a complete validation strategy.
</Callout>

## Performance Benefits

Why validate early?

```python title="Without validation"
@app.get("/matches/{match_id}")
def get_match_by_id(match_id: int):
    # User sends match_id=-999999
    for match in INITIAL_DATA:  # Loop through entire dataset
        if match.id == match_id:
            return match
    # No match found, return error
    return {"error": "Match not found"}
```

```python title="With validation"
@app.get("/matches/{match_id}")
def get_match_by_id(
    match_id: Annotated[int, Path(ge=1)],
):
    # User sends match_id=-999999
    # FastAPI rejects it immediately — function never runs
    # No database lookup, no loop, no wasted CPU
    pass
```

<Callout type="info" title="Fail Fast">
  Validation errors are cheap — checking `ge=1` is a single comparison. Database lookups, API calls, and complex filtering are expensive. By rejecting bad input immediately, you save server resources and give users faster error responses.
</Callout>
